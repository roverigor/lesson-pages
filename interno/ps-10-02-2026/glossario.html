Setup e Configuração

AIOS
Setup
AI-Orchestrated System (Sistema Orquestrado por IA). Framework que coordena múltiplos agentes de IA para gerenciar projetos de desenvolvimento de software completos.
Contexto de Uso
Instalado via aios install. Estrutura com .aios-core/, .aios/, .claude/commands/. Cada projeto precisa de sua própria instalação AIOS com CLAUDE.md específico.
CLAUDE.md
Setup
Arquivo de configuração que atua como "cérebro" do sistema AIOS. Contém regras, heurísticas e contexto específico do projeto.
Contexto de Uso
CRÍTICO: Deve ser POR PROJETO, nunca global. CLAUDE.md global = "frankenstein" (quebra contexto). Cada projeto precisa de seu próprio CLAUDE.md com regras específicas daquele código/arquitetura.
Workspace Master
Setup
Pasta raiz que contém múltiplos projetos individuais. Pode ter seu próprio CLAUDE.md com heurísticas transversais (design system comum, padrões globais).
Contexto de Uso
Estrutura: ~/code/ (workspace master) → projeto-api/, projeto-camban/, projeto-loja-bio/ (projetos individuais com AIOS próprio).
Squad Creator
Setup
Template/framework criado por Alan Nicolas para inicializar projetos com AIOS. Inclui estrutura de agentes, workflows e configurações pré-definidas.
Contexto de Uso
Disponível no GitHub oficial + grupo WhatsApp. Versão mencionada: 5-2-26. Atualizado frequentemente por Alan para necessidades avançadas.
Expansion Packs
Setup
Módulos/extensões do AIOS que adicionam funcionalidades específicas (ex: creator-os, mmos, innerlens). Ficam em pasta separada da estrutura core.
Contexto de Uso
Estrutura: .aios-core/ (framework base) vs expansion-packs/ (extensões customizadas). Sincronizados automaticamente para .claude/commands/ via pre-commit hook.

Workflow de Desenvolvimento

PRD (Product Requirements Document)
Workflow
Documento de Requisitos do Produto. Especificação detalhada de features, regras de negócio e comportamentos esperados. Base para desenvolvimento.
Contexto de Uso
Pode ser PRD completo (do zero ao MVP) ou PRD incremental (feature por feature). Workflow recomendado: Specs modulares → AIOS consolida PRD → Épicos/Stories.
Épico (Epic)
Workflow
Camada grande de funcionalidade dentro do PRD. Equivalente a "andares do prédio". Contém múltiplas stories relacionadas.
Contexto de Uso
Hierarquia: PRD → Épicos (foundation, auth, features) → Stories (sprints) → Tasks (ações atômicas). Criado pelo Project Manager (@pm).
Story (História)
Workflow
Unidade de trabalho dentro de um épico. Equivalente a "sprint". Implementável em dias/semanas. Contém tasks específicas.
Contexto de Uso
Refinada pelo Scrum Master (@sm) e validada pelo Product Owner (@po). Implementada pelo Developer (@dev). Formato: Story 2.1, Story 3.4, etc.
Task (Tarefa)
Workflow
Ação atômica e específica dentro de uma story. Menor unidade executável. Exemplos: criar componente, escrever teste, configurar endpoint.
Contexto de Uso
Executadas sequencialmente pelo agente @dev. Rastreadas via checkboxes em arquivos .md das stories. Comandadas via *task no AIOS.
Doc Specs (Specs Modulares)
Workflow
Documentos de especificação separados por dimensão (UI, Tech, Features, Wireframes). Alternativa a PRD gigante monolítico.
Contexto de Uso
Workflow João: Specs separadas (UI/UX, Tech Stack, Features, Philosophy, Wireframes, Design System) → AIOS consolida PRD. Evita sobrecarga cognitiva da LLM.
Prova de Fogo
Workflow
Etapa de validação onde você "destrói sua ideia". Identifica pontos cegos, falhas de lógica, SWOT, o que NÃO faz sentido.
Contexto de Uso
Workflow: Descarrego → Briefing → Pesquisas → Enriquecer → PROVA DE FOGO → Ajustar → Specs → PRD. Previne desperdício de tempo com ideias inviáveis.
PM (Project Manager)
Workflow
Agente AIOS responsável por criar épicos a partir do PRD. Divide o projeto em camadas grandes de funcionalidade.
Contexto de Uso
Chamado via @pm no Claude Code. Primeiro agente acionado após PRD pronto. Output: Estrutura de épicos (foundation, features, auth, etc.).
PO (Product Owner)
Workflow
Agente AIOS responsável por validar checklists e histórias. Garante que implementação está alinhada com requisitos de negócio.
Contexto de Uso
Chamado via @po. Executa checklist de validação após PM e antes/depois do SM. Valida stories antes de ir para Dev.
SM (Scrum Master)
Workflow
Agente AIOS responsável por refinar histórias (stories) dentro dos épicos. Quebra épicos em sprints implementáveis.
Contexto de Uso
Chamado via @sm. Atua após PO executar checklist e antes do Dev implementar. Output: Stories refinadas com tasks claras.
Dev (Developer)
Workflow
Agente AIOS responsável por implementar código. Executa tasks dentro de stories validadas pelo PO.
Contexto de Uso
Chamado via @dev. Último agente na cadeia (PM → PO → SM → PO → Dev). Implementa código, testes, documenta, faz commits.

Ferramentas e IDEs

Claude Code
Ferramentas
Motor de IA conectado ao terminal. Interface de linha de comando para conversar com LLM (Claude). Roda workflows, agentes e comandos AIOS.
Contexto de Uso
NÃO é uma IDE. É o cérebro conversacional. Pode rodar dentro de IDE (terminal integrado) ou terminal puro. Analogia: Claude Code = motor, IDE = visualização.
IDE (Integrated Development Environment)
Ferramentas
Ambiente visual para ver arquivos, pastas e código. Exemplos: Cursor, AntiGravity, VSCode, Windsurf. "Obsidian do código".
Contexto de Uso
Iniciantes: IDE + terminal integrado rodando Claude Code. Avançados (4+ terminais): terminal puro (mais leve, escalável). IDE ≠ Claude Code.
Deep Serve
Ferramentas
Ferramenta de pesquisa profunda na internet. Busca contexto rico sobre mercado, concorrentes, tecnologias, gaps.
Contexto de Uso
Workflow: Deep Serve/Perplexity → pesquisa de mercado → estruturar projeto → jogar pro AIOS criar PRD. Traz tech stack e lacunas que você não conhecia.
Perplexity
Ferramentas
Motor de busca com IA que retorna respostas contextualizadas. Alternativa ao Deep Serve para pesquisa de mercado e validação de ideias.
Contexto de Uso
Usado antes de criar PRD para identificar concorrentes, tendências, bibliotecas, APIs disponíveis. Enriquece briefing com dados reais.
Architect Agent
Ferramentas
Agente customizado (Google AI Studio) criado por João. Transforma briefing em sitemap, wireframes, estrutura de navegação e fluxos.
Contexto de Uso
Usado ANTES de codar. Input: Briefing. Output: Wireframes + Fluxos + Sitemap. Dá tangibilidade visual à ideia antes de entrar no AIOS.
Cursor
Ferramentas
IDE (fork do VSCode) com IA integrada. Uma das opções de ambiente de desenvolvimento recomendadas para uso com AIOS.
Contexto de Uso
Alternativas: AntiGravity (usado por João), VSCode, Windsurf. Todos funcionam com Claude Code no terminal integrado.
@aios-master
Ferramentas
Agente mentor do AIOS. Responde perguntas sobre processo, metodologia, boas práticas. "Professor pessoal" que ensina tudo sobre AIOS.
Contexto de Uso
Setup recomendado: Janela 1 = @aios-master (só perguntas), Janela 2 = projeto (trabalho). Contextos separados. Pergunte TUDO pra ele.

Metodologias e Frameworks

5 Porquês
Metodologias
Técnica de descoberta de motivações profundas. Perguntar "por quê?" 5 vezes seguidas até chegar ao Nível 5 (liberdade, autonomia, propósito).
Contexto de Uso
Usado para conectar features técnicas a benefícios emocionais. Ex: Tokens = infraestrutura (técnico) → Liberdade geográfica/temporal (emocional).
GPS Framework
Metodologias
Framework de avaliação pedagógica com 10 aspectos: Goal, Semiótica, Engajamento, Domínio, Steps, Personalização, Clareza, Aplicabilidade, Inovação, Retenção.
Contexto de Uso
Usado em avaliações internas de aulas/conteúdo. Notas de 0-10 em cada aspecto. Média final indica qualidade pedagógica.
Metodologia Lendária
Metodologias
Framework pedagógico com 7 pilares para criar conteúdo educacional de alta qualidade. Inclui regras como limite de 2-3 conceitos por aula.
Contexto de Uso
Usado junto com GPS Framework em avaliações. Pilar 3 (Regra de Ouro): máximo 2-3 conceitos por aula para evitar sobrecarga cognitiva.
Matriz RICE
Metodologias
Framework de priorização de features. RICE = Reach (alcance), Impact (impacto), Confidence (confiança), Effort (esforço).
Contexto de Uso
Usado por PM/PO para decidir quais features implementar primeiro. Calcula score: (Reach x Impact x Confidence) / Effort.
Brownfield
Metodologias
Projeto existente (código legado). Adicionar features/melhorias em sistema já funcionando. Oposto de Greenfield (do zero).
Contexto de Uso
PRDs incrementais são ideais para Brownfield. Adicionar 3 telas + OAuth em sistema existente vs. reconstruir tudo.
Greenfield
Metodologias
Projeto do zero. Sem código legado, sem restrições técnicas. Liberdade total de arquitetura e stack.
Contexto de Uso
PRD completo (foundation → MVP) é ideal para Greenfield. AIOS cria épicos de setup, database, auth, features, deploy.

APIs e Integrações

Evolution API
APIs
API não-oficial para WhatsApp. Permite criar instâncias, vincular webhooks, processar mensagens, gerar QR codes. Mais simples que Meta Business API.
Contexto de Uso
Usado por Zé Carlos com Claude Code. Desafio: bloqueios do WhatsApp. Solução: rotação entre 5 instâncias. Começar com Evolution antes de Meta Business.
Meta Business API
APIs
API oficial da Meta (Facebook/Instagram/WhatsApp). Requer aplicativo no Facebook Developers, solicitação de permissões, aprovações burocráticas.
Contexto de Uso
Burocrática mas segura. Cliente precisa aprovar várias etapas. NÃO usar API não-oficial (risco de ban permanente). Para Ads: ÚNICO caminho recomendado.
MCP (Model Context Protocol)
APIs
Protocolo para conectar LLMs a ferramentas externas. Permite que Claude acesse APIs, bancos de dados, ferramentas via interface padronizada.
Contexto de Uso
Usado pelo AIOS para integrar n8n, Playwright, Desktop Commander, Supabase, etc. Cada MCP = ponte entre Claude e ferramenta específica.
Tool Discovery
APIs
Processo pelo qual AIOS identifica e lista MCPs (ferramentas) disponíveis no ambiente. Permite que agentes saibam quais capacidades têm.
Contexto de Uso
AIOS escaneia .mcp.json e diretórios de tools. Agentes descobrem dinamicamente quais APIs podem usar (ex: n8n pra automação, Playwright pra web).
Upstream
APIs
Repositório Git original de onde o projeto foi forkado. Usado para sincronizar atualizações do projeto base para seu fork.
Contexto de Uso
Comando: git atualizar (alias para fetch upstream + merge upstream/main). Mantém fork sincronizado com repositório original.
