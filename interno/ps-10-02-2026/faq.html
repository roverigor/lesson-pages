Setup e Configuração (9 perguntas)

P1: Qual a última versão do Squad Creator? Onde está? ▼
Resposta: GitHub oficial + grupo WhatsApp. Versão: 5-2-26.

O Squad Creator do Alan Nicolas está disponível no repositório oficial do GitHub e as atualizações são compartilhadas no grupo do WhatsApp. A versão mencionada durante o PS foi "5-2-26 squad-creator 3h33". Recomenda-se atualizar via git pull do GitHub.
P2: Copiar pasta AIOS ou instalar do zero em projeto novo? ▼
Resposta: Ambos funcionam. Copiar: toda infraestrutura EXCETO CLAUDE.md. Instalar: claude init + aios install.

Opção 1 - Copiar:
• Copiar .aios-core/, .aios/, .claude/commands/
• IMPORTANTE: NÃO copiar CLAUDE.md (específico do projeto original)
• Colar na pasta exatamente espelhado

Opção 2 - Instalar do zero:
• Rodar claude init na pasta nova
• Rodar aios install para configurar

Regra de ouro: Cada projeto precisa de seu próprio CLAUDE.md (cérebro do sistema). Se copiar CLAUDE.md de outro projeto → vai quebrar (contexto errado).
P3: Como atualizar AIOS em todos os projetos? ▼
Resposta: Rodar aios install em cada repositório/pasta.

1. Ir em cada repositório/pasta onde AIOS está instalado
2. Rodar aios install ou aios update
3. Atualiza automaticamente em cada projeto

Dica Pro: Se você tem vários projetos, vale ter um workspace central. O workspace opera outras pastas filhas, concentrando atualizações.
P4: CLAUDE.md por projeto ou global? ▼
Resposta: Por projeto. CLAUDE.md global = frankenstein (quebra contexto).

O ideal é por PROJETO:
• CLAUDE.md = cérebro do sistema, primeira coisa carregada
• CLAUDE.md "frankenstein" (misturando vários contextos) → confunde a LLM
• Cada projeto tem contexto, arquivos, pastas específicas
• CLAUDE.md global não sabe quais arquivos existem em cada projeto

Exceção: Você pode ter um CLAUDE.md MASTER (na raiz do workspace) com heurísticas transversais (design system comum). Cada projeto ainda tem seu próprio CLAUDE.md específico.
P5: Como organizar workspace vs projetos? ▼
Resposta: Workspace master na raiz + projetos individuais com AIOS próprio. Cada um com seu CLAUDE.md.

Estrutura recomendada:
~/code/ (Workspace master)
├── .claude/CLAUDE.md (CLAUDE.md MASTER - heurísticas globais)
├── projeto-api/ (Projeto 1)
│ ├── .claude/CLAUDE.md (CLAUDE.md específico)
│ ├── .aios-core/
│ └── .aios/
├── projeto-camban/ (Projeto 2)
└── projeto-loja-bio/ (Projeto 3)

Pastas auxiliares (separadas):
~/marketing/ (agentes marketing)
~/mentes-clonadas/ (mentes dos mentores)
P6: Pastas auxiliares (agentes customizados) - onde ficam? ▼
Resposta: Fora do workspace (ex: ~/marketing/, ~/mentes-clonadas/). Copiar pra dentro do projeto quando precisar.

Workflow:
1. Abrir IDE (Cursor, AntiGravity, VSCode) na pasta do projeto
2. Dentro da IDE, usar terminal com Claude Code
3. Quando precisar de agente externo (ex: UX master) → copiar pra dentro do projeto

Mantém separação de contextos e permite reutilização de agentes entre projetos.
P7: Como usar Deep Serve pra pesquisa de mercado? ▼
Resposta: Cloud AI Studio → pesquisa de mercado → estruturar projeto → jogar pro AIOS. Traz gaps, concorrentes, tech stack.

Processo:
1. Abrir Cloud AI Studio (ou Google AI Studio)
2. Fazer pesquisa de mercado: "Pesquise sobre [seu nicho/produto]"
3. Busca na internet inteira, retorna contexto rico
4. Estruturar projeto: "Com base nessa pesquisa, estruture um projeto para implementar X usando AIOS"
5. Jogar resultado pro AIOS pra criar PRD

Por que funciona: Deep Serve traz gaps que você não pensou, identifica concorrentes, sugere tech stack (bibliotecas que você nem conhecia), direciona features.
P8: Como trabalhar com múltiplas janelas Claude Code? ▼
Resposta: Janela 1: @aios-master (só perguntas). Janela 2: projeto (desenvolvimento).

Setup recomendado:

Janela 1 - AIOS Master (Perguntas):
• Terminal aberto só com @aios-master
• Usar pra fazer TODAS as perguntas
• "Como faço X?", "O que é Y?", "Qual próximo passo?"
• Ele ensina tudo

Janela 2 - Desenvolvimento:
• Terminal do projeto
• Onde você roda agentes (@dev, @po, @sm, etc.)
• Onde acontece o trabalho

Vantagens: Contextos separados, histórico de perguntas não polui histórico de dev, AIOS Master vira seu mentor pessoal.
P9: Architect Agent - o que é? ▼
Resposta: Agente do João (Google AI Studio). Input: briefing. Output: sitemap, wireframes, estrutura navegação, fluxos. Dá tangibilidade antes de codar.

Como funciona:
• Agente customizado criado no Google AI Studio
• Input: Briefing do projeto
• Output: Sitemap completo + Wireframes + Estrutura de navegação + Fluxos de usuário
• Usado ANTES de começar a codar

Workflow de Desenvolvimento (8 perguntas)

P10: Fazer tudo fora do AIOS e trazer depois? ▼
Resposta: Sim (preferência do João).

Workflow completo:
1. Sessão descarrego (áudio/call)
2. Transformar em briefing
3. Pesquisas (mercado + tech)
4. Enriquecer briefing
5. Prova de fogo (destrua ideia)
6. Enriquecer novamente
7. Doc specs modularizadas (UI, tech, wireframes, features)
8. AIOS consolida PRD

Ferramenta favorita: Architect Agent (Google AI Studio) para criar sitemap, wireframes e fluxos antes de codar.
P11: PRD inteiro ou só feature? ▼
Resposta: Ambos. Incremental (feature por feature) é melhor que PRD gigante.

Opção A - PRD do Zero:
• Do zero até MVP
• Grandes épicos, muitas stories

Opção B - PRD Incremental (Recomendado):
• Produto já existe
• PRD só pra adicionar 3 novas telas + 3 features
• Épicos menores, menos stories

Vantagens do incremental: Menos pontos de conexão = menos chance de erro probabilístico. LLMs são voláteis - quanto maior o processo, maior risco.
P12: Fluxo recomendado depois do PRD? ▼
Resposta: PM → PO → checklist → SM → PO valida → Dev implementa.

Fluxo detalhado:
PRD pronto
↓
Project Manager (@pm) - Criar épicos
↓
Product Owner (@po) - Executar checklist de validação
↓
Scrum Master (@sm) - Refinar histórias (stories)
↓
Product Owner (@po) - Validar tudo de novo
↓
Developer (@dev) - Implementar
P13: Por que modularizar specs ao invés de PRD gigante de uma vez? ▼
Resposta: LLM tem 100% capacidade num ciclo. PRD gigante = prova de vestibular gigante = cansaço no final. Specs modulares = laser focus = qualidade.

Por quê modularizar:
• LLM tem 100% capacidade cognitiva num ciclo input→output
• PRD gigante = prova de vestibular gigante = LLM cansada no final
• Specs modulares = laser focus = 100% capacidade em cada spec

Specs recomendadas:
• Doc Spec: UI/UX
• Doc Spec: Filosofia/Conceito
• Doc Spec: Tech Stack (bibliotecas, APIs, integrações)
• Doc Spec: Wireframes + Fluxos de usuário
• Doc Spec: Features
• Doc Spec: Design System
P14: Como validar ideia antes de codar? ▼
Resposta: Prova de fogo - "Destrua sua ideia". Ponto cego, SWOT, o que NÃO faz sentido.

Processo Prova de Fogo:
1. Descarrego → Briefing
2. Pesquisas (mercado + tech)
3. Enriquecer briefing
4. PROVA DE FOGO: "Destrua sua ideia"
5. Perguntar: Pontos cegos? SWOT? O que NÃO faz sentido? Onde estou viajando?
6. Ajustar briefing com insights
7. Specs → PRD

Previne desperdício de tempo com ideias inviáveis.
P15: Diferença entre IDE e Claude Code? ▼
Resposta: IDE = Obsidian do código (visualizar arquivos). Claude Code = motor IA no terminal. São coisas diferentes.

IDE (Integrated Development Environment):
• Ambiente visual pra ver arquivos, pastas, código
• Exemplos: Cursor, AntiGravity, VSCode, Windsurf
• "Obsidian do código"

Claude Code:
• Motor de IA conectado ao terminal
• Roda dentro do terminal (pode ser dentro da IDE ou terminal puro)
• É o cérebro que conversa com você

Preferência João: IDE (AntiGravity) + Claude Code no terminal integrado. Vantagens: visualização de pastas, controle de versionamento (git), ver o que tá sendo editado.
P16: IDE ou terminal puro? ▼
Resposta: Iniciantes: IDE + terminal integrado. Avançados rodando 4+ terminais: terminal puro (mais leve, escalável).

Quando usar terminal puro (sem IDE):
• Quando rodar 4+ terminais simultâneos
• IDE começa a sobrecarregar
• Poluição visual com muitos terminais
• Terminal puro é infinitamente escalável (20+ tabs sem travar)

Dica pra iniciantes: Começar com IDE + terminal integrado. Melhor visualização, mais fácil se achar. Depois migrar pra terminal puro se quiser escalar.
P17: Como saber se estou no caminho certo? ▼
Resposta: Pergunta pro @aios-master. Ele ensina tudo. Janela separada só pra perguntas.

Use o @aios-master como seu mentor pessoal. Configure uma janela separada só pra perguntas e pergunte TUDO: "Como faço X?", "O que é Y?", "Qual próximo passo?", "Estou no caminho certo?"

Ele tem todo o conhecimento do AIOS embutido e pode guiar você através de qualquer processo.

Casos de Uso / Negócios (6 perguntas)

P18: Como aplicar AIOS em consultoria jurídica? ▼
Resposta: Cliente com 30-50+ clientes, não escala. AIOS pra criar sistemas de consistência operacional, identificar gargalos. Desafio: ranço com IA (funcionários usaram errado).

Problema:
• Cliente com 30-50+ clientes (triplicou)
• Não consegue dar qualidade de atendimento
• Processos empíricos dependentes dos sócios
• Gargalos em várias áreas (financeiro, jurídico, RH)

Solução proposta:
• Criar sistemas com AIOS para consistência operacional
• Identificar gargalos que impedem escala
• Implantar IA nos processos do cliente

Desafio cultural: Funcionários usando IA errado (delegando decisões em vez de agilizar processos). Quebrar barreira do ranço com IA criado por mal uso.
P19: Como aplicar AIOS em saúde masculina (urologia)? ▼
Resposta: IA como primeiro atendimento anônimo (reduz vergonha), acompanhamento contínuo, CS da IA 24/7. Desafio: quadramento (monetização) complexo, precisa minerar mais.

Problema:
• Homem historicamente não cuida da saúde
• Barreira cultural pra falar sobre saúde masculina/urologia
• Atendimento episódico vs. acompanhamento contínuo

Oportunidades com IA:
• IA como primeiro atendimento anônimo (pessoas preferem falar com IA do que com médico sobre certos assuntos - reduz vergonha)
• Acompanhamento contínuo (ex: reação alérgica à meia-noite → CS da IA responde)
• Agilizar triagem e diagnóstico inicial

Desafio: Solução tem potencial MAS quadramento (monetização) é complexo. Precisa minerar mais pra entender onde encaixa no processo.

P21: WhatsApp: Evolution API ou Meta Business API? ▼
Resposta: Começar Evolution (simples). Meta Business é burocrático (cliente precisa aprovar várias coisas, você pega na mão).

Evolution API (não-oficial):
• Mais simples pra começar
• Documentação clara
• Risco de bloqueios
• Boa pra aprender e testar

Meta Business API (oficial):
• Burocrático
• Cliente precisa aprovar várias etapas
• Você tem que pegar na mão do cliente
• Pessoas cobram só pra fazer cadastro na Meta
• Quem gira muito tráfego sofre com Meta Business
• Seguro e oficial

Recomendação: Começar com Evolution pra aprender. Migrar pra Meta Business se precisar escalar com segurança.

Mindset e Práticas (5 perguntas)

P24: Como não ficar preso esperando próxima versão do Squad Creator? ▼
Resposta: Alan lança versões pra necessidades dele (nível avançado). Se você não sente necessidade → versão atual serve. 80/20 = dominar PROCESSO, não ferramenta.

80/20 do aprendizado:
• Não é dominar ferramentas
• Não é acompanhar todas atualizações do Alan
• É dominar o PROCESSO (como tudo se conecta, lógica por trás, motor do sistema)

Por quê Alan lança tantas versões:
• Ele tá num nível avançado
• Precisa de funcionalidades que você ainda não precisa
• Se você não sente necessidade de atualização → versão atual serve perfeitamente

Conselho: Pegue versão atual, salve numa pasta, COMECE A DESENVOLVER. Se bloquear, atualiza. Se não bloquear, segue com o que tem.
P25: E se não sair perfeito na primeira vez? ▼
Resposta: Faz de novo. Conhecimento empírico > teórico. Progresso incremental (hoje melhor que ontem).

Mindset correto:
• FAZ DE NOVO
• Não importa o app que vai sair
• O que importa é dominar o processo
• Conhecimento empírico > conhecimento teórico

Exemplo (Torriani):
"Fiz tudo no Cloud, achei que nunca ia conseguir no Claude Code. Hoje tá tudo no Claude Code, não quero nem olhar pro Cloud. Progresso incremental: Hoje melhor que ontem."

Conselho: Não tenha medo de começar do zero de novo. Pegue processo que João ensinou. Crie qualquer coisa (lista de tarefas, organizador de notas). Teste o processo. Vai aprender FAZENDO.
P26: Como começar se estou perdido? ▼
Resposta: Crie QUALQUER COISA simples (lista de tarefas, organizador de notas). Siga workflow do João. Teste processo. Aprende FAZENDO.

Passo a passo pra começar:

Não existe conhecimento mais valioso que conhecimento empírico. Campo de batalha, skin in the game. 80/20: Colocar mão na massa.
P27: Devo me preocupar com atualizações constantes? ▼
Resposta: Não. Se tá funcionando, não precisa atualizar. Atualiza só se bloquear.

Se sua versão do Squad Creator tá funcionando, não precisa atualizar. Atualiza só se você:
• Bloquear em alguma funcionalidade
• Encontrar bug crítico
• Precisar de feature específica da nova versão

Regra de ouro: "Se tá funcionando, não mexe." Foque em dominar o processo, não em ter a última versão da ferramenta.
P28: Quanto tempo ficar planejando antes de codar? ▼
Resposta: Não ficar eternamente. Fazer prova de fogo, validar, e IR PRA PRÁTICA. Senão fica só na ideação.

Planejamento ideal:
1. Descarrego (1 sessão)
2. Briefing (1-2 horas)
3. Pesquisas (2-4 horas)
4. Prova de fogo (1 hora)
5. Specs modulares (4-8 horas)
6. IR PRA PRÁTICA (não esperar mais)
