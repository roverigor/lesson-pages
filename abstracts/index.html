Como Criar Squads de IA que Se Atualizam Sozinhos + Tool Discovery + Modelos Mentais para Priorização

Resumo

Esta foi uma aula preparatória intensiva para o Cohort AIOS Avançado, onde Alan Nicolas e Pedro Valerio Lopez mostraram técnicas avançadas de automação e arquitetura de sistemas de IA.

Tema Central: Como construir sistemas que se auto-melhoram através de Tool Discovery automático e priorização inteligente usando frameworks comprovados.

Demonstrações Práticas: Configuração de Upstream para manter AIOS atualizado automaticamente, integração completa com Google Gemini CLI (lançado recentemente), sistema de Tool Discovery que pesquisa MCPs, Skills, APIs e projetos Github, e aplicação da Matriz RICE para decidir quais tools implementar.

Insights Profundos: Pedro revelou sua estrutura de priorização automática usando Matriz RICE no ClickUp. Alan demonstrou clonagem de mentes (colocou clone do Pedro no Squad Creator). Discussão sobre monorepo vs múltiplos repositórios e como usar Google App Script para automação em planilhas (caso real do Pedro).

Mensagem Principal: "A diferença entre quem tem resultado não é conhecimento, mas executar rapidamente. Mesmo se não entender tudo, vá fazendo — seu aprendizado vem da ação, não da teoria."

Aula 01 04 Fev 2026

AIOS Avançado: Arquitetura, Workflows e Mindset Para Desenvolvimento 10x Mais Rápido

Resumo

Nesta aula de 3h10min, Alan Nicolas e Pedro Valerio Lopez desmistificaram a arquitetura completa do AIOS Framework e transformaram a forma como você deve pensar desenvolvimento.

Mentalidade Primeiro: Alan começou com um choque de realidade — tokens NÃO são luxo, são infraestrutura. Assim como sua empresa precisa de energia elétrica e internet, ela precisa de tokens para sobreviver em 2026. Ele mostrou que 1 mês do trabalho dele equivale a R$1M em custos de time tradicional, enquanto gasta R$200-500 em tokens.

Arquitetura Revelada: Pedro desmontou o AIOS peça por peça. .aios-core é o framework base (nunca toque!), expansion-packs são suas customizações, e claude.md é a "Terra" — regras globais. Monorepo significa tudo na mesma pasta, compartilhando configuração.

Epic → Story → Task: A hierarquia de trabalho foi o ponto de virada. Epic = grande objetivo, Story = pedaço entregável, Task = ação específica. Workflows orquestram Tasks automaticamente. Essa decomposição é o que permite AIOS não se perder em contexto gigante.

@ vs / — Revelação Crítica: @ simula um agente (persona-switching), / spawna agente DE VERDADE. @pm apenas muda tom de voz, /pm executa workflow completo com ferramentas.

BOB (Baby On Board): Agente para iniciantes — modo com "rodinhas" que guia você passo a passo. Alan e Pedro enfatizaram: não tenha vergonha de usar BOB.

Documentação > Conversa: O momento "mind blowing" da aula. Criar DOCUMENTOS estruturados funciona MELHOR que chat conversacional. Documentos dão contexto claro e determinismo. Chats levam a ambiguidade e erros.

Ferramentas do Ecossistema: CodeRabbit (code review IA), GitHub Actions (CI/CD), Environment Bootstrap (instala tudo), YOLO Mode (vai fundo sem perguntar).

Tópicos Abordados
Mindset sobre custo de tokens (investimento, não gasto) Estrutura completa do AIOS (.aios-core, expansion-packs, claude.md) Hierarquia Epic → Story → Task → Workflow Diferença entre @ (persona-switching) e / (spawn de agente) Agente BOB (Baby On Board) para iniciantes CodeRabbit, GitHub Actions, Environment Bootstrap Documentação > Conversa (determinismo) Brownfield vs Greenfield Monorepo: múltiplos projetos na mesma pasta

Takeaways

Mude sua mentalidade sobre tokens: Assine Claude Pro ou API. Trate como conta de luz, não como luxo. Sem infraestrutura adequada, você não pratica corretamente.

Explore a estrutura do AIOS: Abra .aios-core e entenda as pastas (agents, tasks, workflows). Leia claude.md completo do seu projeto. Identifique expansion-packs que você usa.

Pratique Epic → Story → Task no papel: Pegue um projeto real e quebre em 1 Epic, 3-5 Stories e 2-3 Tasks por Story. NÃO execute no AIOS ainda — apenas pratique PENSAR nessa estrutura.

Use BOB: Chame /bob ou agente Baby On Board para criar algo simples. Deixe ele te guiar antes de ir pro modo avançado.

Documente antes de executar: Crie brief.md ou prd.md com o que quer, estruture Epic → Stories no documento, só então chame AIOS para executar. Documentação > Conversa = determinismo > ambiguidade.

Instale CodeRabbit e rode /devops: Crie conta gratuita, no seu projeto rode /devops e execute *environment-bootstrap. Conecte GitHub e rode primeiro code review.

Aula 02 11 Fev 2026

Squad Creator: Construindo Sistemas de Agentes Inteligentes com Economia de Tokens

Resumo

Esta aula apresentou o Squad Creator v3.0-v4.0, uma ferramenta revolucionária para construir sistemas de agentes AI com economia de tokens dramática. O instrutor demonstrou como alcançar redução de 90% nos custos de LLM ao usar Haiku ($0.25 por 1M tokens) em vez de Opus ($15 por 1M tokens), mantendo ou até superando a qualidade dos outputs através de técnicas de prompt engineering e model qualification.

A sessão começou com instalação hands-on do Squad Creator via npm, seguida de demonstração do dashboard interativo onde é possível visualizar agentes, workflows e métricas de execução. Um dos conceitos mais poderosos apresentados foi o "validate squad" — filosofia de testar sistemas de agentes em escala pequena antes de construir aplicações completas, economizando tempo e dinheiro.

O instrutor enfatizou a importância de operações determinísticas: converter tarefas que não precisam de raciocínio LLM em scripts simples. Por exemplo, validação de formato YAML não precisa custar $0.03 por chamada quando um script Python pode fazer isso instantaneamente. Esta abordagem híbrida (scripts + LLM apenas quando necessário) foi demonstrada em pipelines ETL reais.

A integração com Open Router permite usar múltiplos provedores de LLM (OpenAI, Anthropic, Cohere, etc) com fallback automático e rate limiting inteligente. O sistema de Model Tier Qualification classifica automaticamente tarefas em Tier 1 (Haiku), Tier 2 (Sonnet) ou Tier 3 (Opus) baseado em complexidade, mas permite override manual quando o desenvolvedor conhece melhor o contexto.

Pontos-Chave

[00:23] Haiku consegue qualidade superior ao Opus em muitas tarefas quando o prompt é bem construído, custando 60x menos. — Demonstração comparativa de outputs com mesmos prompts em ambos modelos.

[00:45] "Valide seus squads antes de construir a aplicação. É muito mais barato testar com 10 exemplos do que descobrir problemas em produção." — Alan Nicolas, filosofia de validação antes de escalar.

[01:12] Model Tier Qualification: Sistema automático que classifica tarefas por complexidade e atribui o modelo mais econômico capaz de resolver. — Reduz custos em 70-90% sem perder qualidade.

[01:34] "Operações determinísticas devem SEMPRE ser scripts, não chamadas LLM. Ex: validar JSON, contar palavras, extrair timestamps." — Alan Nicolas. Zero custo vs $0.01-0.05 por operação.

[02:05] "Open Router é como ter um seguro: se a OpenAI cair, você automaticamente faz fallback para Anthropic sem mudar uma linha de código." — Rodrigo, sobre resiliência com multi-provider.

Takeaways

Comece com Validação, Não com Implementação: Antes de construir uma aplicação com agentes, crie um squad mínimo, teste com 10-20 exemplos, meça qualidade e custo. Só escale quando validar. Ferramenta: squad validate + dashboard de métricas.

Use Haiku como Default, Opus como Exception: Configure Haiku como modelo padrão para todas as tarefas. Identifique quais realmente precisam de Opus (normalmente <10%). Meça a diferença de qualidade objetivamente. Economia esperada: 70-90%.

Scripts > LLM para Operações Determinísticas: Liste todas as operações do seu workflow. Para cada uma, pergunte: "isso precisa de raciocínio ou é algorítmico?" Converta algorítmicas em scripts Python/JS. Ex: validação de formato, parsing estruturado, contagens, extrações com regex.

Integre Open Router para Resiliência: Configure fallback entre provedores. Uma API key do Open Router dá acesso a 20+ provedores. Configuração em 10 minutos que evita horas de downtime.

Squad Creator Pro: Como Criar Squads Efetivos com Tiers Estratégico, Tático e Operacional

Resumo

A Aula 03 do Cohort AIOS Avançado foi dedicada inteiramente ao Squad Creator Pro, a ferramenta central do AIOS para criação de times de agentes de IA. Com duração de aproximadamente 4 horas e 40 minutos, a aula combinou teoria profunda sobre arquitetura de squads com demonstrações práticas ao vivo.

Alan Nicolas abriu a aula mostrando o novo dashboard de observabilidade que está construindo, incluindo visualização de squads, agentes, Kanban de stories, monitor em tempo real e a nova funcionalidade de renderização visual de workflows como diagramas. Explicou a filosofia "CLI First" — quanto menos UI, mais poder a IA tem — e como a observabilidade é secundária em relação ao trabalho real feito via terminal.

O tema central foi a transição de Commands para Agents e Skills, um movimento que alinha o AIOS com a direção de todas as LLMs. Pedro Valério detalhou a importância do front matter nos agentes e como isso afeta carregamento de contexto e consumo de tokens. A migração levou 5 horas de processamento contínuo no Codex e prepara o sistema para o futuro Agent Teams do Claude Code.

Pedro introduziu o conceito revolucionário dos três Tiers de squads: Estratégico (C-levels, visão e regras), Tático (gestores, processos e templates) e Operacional (execução). Ele enfatizou que a maioria das empresas e usuários pula o tier tático, criando um gap de regras que impede consistência nos outputs. O Squad Creator Pro já define automaticamente os três tiers ao criar qualquer squad.

Alan demonstrou ao vivo a criação completa de um squad usando uma ideia do aluno Adriano: "Squad de extração de regras de negócios de sistemas legados". O processo incluiu pesquisa automática de especialistas, filtragem pelo advogado do diabo, definição de tiers, criação paralela de agentes, e validação com score numérico. Também mostrou como usa seu clone (Alan Nicolas) para pesquisar livros específicos dos especialistas selecionados, maximizando a qualidade do squad.

Um dos momentos mais impressionantes foi a demonstração do Advisory Board virtual, onde clones de Peter Thiel, Naval Ravikant, Derek Sivers, Charlie Munger, e os próprios clones de Alan e Pedro debateram sobre a estratégia de criação de um squad. O clone do Pedro reproduziu fielmente seu estilo de pensamento focado em processo e determinismo, enquanto o clone do Alan trouxe a visão de curadoria e exploração semântica.

Pedro apresentou uma visão sofisticada de squads como BUs (Business Units) independentes, cada uma com input e output claros, que podem ser produtizadas separadamente. Ele mostrou exemplos reais da sua empresa Uflows, onde squads de Copy, Produção, GenAI e Edição funcionam como BUs encadeadas, cada uma recebendo o output da anterior como input.

A aula encerrou com Alan mostrando funções avançadas como *discover-tools (que pesquisa na internet MCPs, APIs e projetos open source para cada agente), Tech Research (skill para deep research técnico), e a visão de futuro com LLM in Sandbox para serviços online. O dever de casa é criar squads, validar com *validate-squad, e usar *discover-tools para enriquecer os squads criados.

Takeaways

Crie seu primeiro squad a partir de uma única frase: Abra o Claude Code, chame /squad-tif, use *create-squad e descreva em uma frase o problema que quer resolver. O Squad Creator Pro com 67 comandos elimina a barreira de entrada para criação de squads complexos.

Sempre valide seus squads com *validate-squad: Após criar um squad, abra uma nova aba e rode *validate-squad com o nome do squad para verificar score e ajustes. Evita usar squads com configurações quebradas ou incompletas que geram outputs ruins.

Pense em squads como BUs (Business Units): Para cada squad que criar, defina: qual é o input macro? Qual é o output macro? Alguém compraria esse serviço? Isso permite produtizar squads e identificar oportunidades de negócio reais.

Não pule o tier tático: Ao planejar seus squads, pergunte: tenho squads táticos que criam as regras que meus squads operacionais seguem? Isso garante consistência nos outputs e permite que operacionais sigam regras claras.

Use Codex para processos longos e testes: Instale o app mobile do Codex. Use para rodar validações, testes longos e criação de squads pesados — tokens praticamente ilimitados eliminam a limitação que trava o trabalho no Claude Code CLI.

Crie squads na área que você tem repertório: Comece criando squads na sua área de domínio. Para áreas desconhecidas, consulte especialistas reais para validar. Evita criar squads bonitos que parecem funcionar mas produzem outputs medíocres.

Use *discover-tools para enriquecer seus squads: Após criar e validar um squad, rode *discover-tools com o nome de cada agente para encontrar MCPs, APIs e projetos open source complementares que amplificam enormemente as capacidades.

Monte seu Advisory Board pessoal: Use o Squad Creator para criar um grupo de advisors personalizado ao seu perfil. Debate ideias com eles antes de executar — acesso a perspectivas diversas de grandes pensadores para tomar decisões melhores.

Token Economy e Custos
Por que Haiku às vezes dá resultado melhor que Opus se é mais barato?
Não é mágica, é prompt engineering: 1. Haiku é mais focado e direto para tarefas estruturadas 2. Opus pode "overengenharia" em tarefas simples 3. Prompts otimizados para Haiku são mais diretos e estruturados 4. Haiku é excelente em seguir formatos (JSON, YAML, listas) Regra prática: Se sua tarefa é "transformar A em B" com regras claras, Haiku provavelmente é melhor E mais barato.

AvançadoAula 03agentsskillsmigração
Por que CLI First e não UI First?
CLI First porque: IA trabalha melhor com Markdown, YAML, JSON. Menos camadas de abstração = mais poder. Cursor (UI) está perdendo para Claude Code (CLI). UI limita porque: cria camadas visuais, restringe opções do modelo, adiciona complexidade. Observabilidade (segundo princípio): Dashboard web para visualizar, Monitor de agentes em tempo real, Skill /squad para resumo textual. Mas tudo é READ-ONLY, não edita pela UI. Como sei se uma operação deve ser script ou LLM call?
Pergunte: "Isso precisa de raciocínio ou é algorítmico?" DEVE SER SCRIPT: Validar JSON/YAML, contar palavras, extrair timestamps, parsing com regex, conversões, ordenação. DEVE SER LLM: Entender contexto semântico, classificar sentimento, resumir, gerar texto. Regra de ouro: Se você consegue escrever if/else ou regex, deve ser script.
