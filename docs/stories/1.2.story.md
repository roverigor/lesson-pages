## Descrição

Como **coordenador da Academia Lendária**, preciso que o sistema processe automaticamente cada notificação inserida no banco, enviando mensagens WhatsApp para grupos de alunos e mentores individuais via Evolution API — com estado rastreável (pending → processing → sent/partial/failed) e auditoria completa de cada envio.

**Contexto:** A Edge Function `send-whatsapp` tem um skeleton em `supabase/functions/send-whatsapp/index.ts`. Esta story implementa a lógica completa: receber o webhook do Supabase, processar a notificação, renderizar o template, enviar via Evolution API e atualizar o state machine.

## Acceptance Criteria

### AC1 — Edge Function recebe e valida o payload do webhook
- **Dado** que o Database Webhook faz POST na Edge Function com `{ type, table, record, schema }`
- **Quando** a function for invocada
- **Então** deve extrair `record` (o registro da tabela `notifications`) e retornar 200 para qualquer payload válido

### AC2 — Guard: apenas processa status `pending`
- **Dado** que uma notificação com status `processing`, `sent` ou `failed` aciona o webhook (edge case)
- **Quando** a function verificar o status
- **Então** deve retornar 200 imediatamente sem qualquer processamento (idempotência)

### AC3 — Transição atômica para `processing`
- **Dado** que uma notificação com status `pending` é recebida
- **Quando** o processamento iniciar
- **Então** deve executar `UPDATE notifications SET status = 'processing' WHERE id = $id AND status = 'pending'` antes de qualquer chamada à Evolution API, prevenindo processamento duplicado em caso de retry do webhook

### AC4 — Template engine funcional
- **Dado** um `message_template` com variáveis `{{class_name}}`, `{{cohort_name}}`, `{{zoom_link}}`, `{{class_time_start}}`, `{{class_time_end}}`, `{{class_weekday}}`, `{{class_professor}}`, `{{class_host}}`, `{{mentor_name}}`, `{{mentors_list}}`
- **Quando** o template for renderizado
- **Então** todas as variáveis presentes no template devem ser substituídas pelos valores buscados das tabelas relacionadas; variáveis sem valor ficam como string vazia (não como `{{variavel}}` literal)

### AC5 — Envio para grupo WhatsApp (target_type = 'group' ou 'both')
- **Dado** uma notificação com `target_type = 'group'` ou `'both'` e `target_group_jid` preenchido
- **Quando** o envio for processado
- **Então** deve fazer `POST /message/sendText/{instance}` na Evolution API com `{ number: target_group_jid, text: message_rendered }` e headers `{ "apikey": EVOLUTION_API_KEY, "Content-Type": "application/json" }`

### AC6 — Envio individual para mentores (target_type = 'individual' ou 'both')
- **Dado** uma notificação com `target_type = 'individual'` ou `'both'` e `class_id` preenchido
- **Quando** o envio for processado
- **Então** deve buscar todos os mentores em `class_mentors` JOIN `mentors` para o `class_id`, e enviar para cada phone individualmente via Evolution API

### AC7 — State machine atualizado corretamente
- **Dado** que o processamento completou
- **Quando** todos os envios forem tentados
- **Então:**
  - Todos enviados com sucesso → `status = 'sent'`, `sent_at = now()`
  - Grupo OK mas algum individual falhou (ou vice-versa) → `status = 'partial'`
  - Todos falharam → `status = 'failed'`, `error_message` preenchido
  - Em todos os casos: `evolution_response` JSONB com a response da API, `processed_at = now()`

### AC8 — Retry logic
- **Dado** uma notificação com `status = 'failed'` e `retry_count < max_retries`
- **Quando** for atualizada para `status = 'pending'` novamente (admin ou processo externo)
- **Então** a function deve processar normalmente incrementando `retry_count`; se `retry_count >= max_retries`, manter como `failed` sem novo processamento

### AC9 — Secrets nunca hardcoded
- **Dado** que `EVOLUTION_API_URL`, `EVOLUTION_API_KEY`, `EVOLUTION_INSTANCE` são dados sensíveis
- **Quando** o código for revisado
- **Então** todos os valores devem ser lidos exclusivamente de `Deno.env.get('NOME_DA_VAR')` — nenhum valor hardcoded no código-fonte

### AC10 — Logs estruturados sem dados sensíveis
- **Dado** que a function executa em produção
- **Quando** eventos importantes ocorrem (início, envio, erro, conclusão)
- **Então** deve logar com `console.log` em JSON estruturado `{ level, notification_id, status, timestamp }` — sem logar phones, JIDs ou conteúdo de mensagens

### Runtime: Deno (Edge Functions)
Edge Functions do Supabase usam Deno. Imports via URL ou `npm:` prefix:
```typescript
import { createClient } from 'npm:@supabase/supabase-js@2'
```

### Supabase client dentro da Edge Function
```typescript
const supabase = createClient(
  Deno.env.get('SUPABASE_URL')!,
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!, // service_role bypassa RLS
)
```

### Template engine — implementação simples
```typescript
function renderTemplate(template: string, vars: Record<string, string>): string {
  return Object.entries(vars).reduce(
    (msg, [key, value]) => msg.replaceAll(`{{${key}}}`, value ?? ''),
    template
  );
}
```

### Evolution API — request
```typescript
const response = await fetch(
  `${EVOLUTION_API_URL}/message/sendText/${EVOLUTION_INSTANCE}`,
  {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'apikey': EVOLUTION_API_KEY,
    },
    body: JSON.stringify({ number: target, text: message }),
  }
);
```

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|-------------|---------|-----------|
| Evolution API inacessível da Edge Function | Média | Alto | VPS em rede pública; testar conectividade antes do deploy |
| Timeout 60s com muitos mentores | Baixa | Médio | A função marca 'processing' antes; falha parcial → status 'partial' |
