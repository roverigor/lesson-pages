## Status
SUPERSEDED-BY-INVESTIGATION

> **Investigation @aiox-master (2026-05-22):** Grep audit T1 confirma que `classes.cohort_id` **NÃO existe** no schema atual:
>
> - Schema baseline `20260402190833_baseline_existing_schema.sql:80-100` define `classes` table SEM coluna `cohort_id`
> - Subsequent ALTER TABLE classes statements adicionam: `type, start_date, end_date, zoom_link, zoom_meeting_id, kind, reminder_enabled` — nenhum `cohort_id`
> - Grep `classes\.cohort_id` em supabase/functions/ + admin/ + js/ = 0 matches
> - Relação class↔cohort vive exclusivamente em `class_cohorts` (M:N) e `class_cohort_access`
>
> **Conclusão:** Premissa da story (drop coluna deprecated) é incorreta. Discovery `02-db-audit.md §3.3` interpretou erroneamente refs a `cohort_id` em outras tabelas (student_cohorts, zoom_meetings, student_nps) como sendo `classes.cohort_id`.
>
> **Decisão:** Story superseded. Não há trabalho técnico necessário — sistema já está no "estado pós-cleanup" descrito. Recomendação: atualizar `docs/discovery/2026-05-22/02-db-audit.md §3.3` removendo afirmação incorreta.
>
> **Sem impacto operacional.** Sistema funciona normalmente via `class_cohorts` JOIN.

## Epic

## Priority
P1 — Queries inconsistentes silenciosas (algumas usam `classes.cohort_id`, outras `class_cohorts`). `get_weekly_attendance()` pode buscar via path errado dependendo de quando query foi escrita.

## Story
Como **administrador da Academia Lendária**,
Eu preciso que **coluna deprecated `classes.cohort_id` seja removida e 100% das queries/RPCs/edge functions usem `class_cohorts` (M:N table)**,
Para que **toda aula seja resolvida via path único e consistente, sem race condition entre paths antigo e novo**.

## Contexto

Discovery `02-db-audit.md` §3.3 identificou **dual-identity** de turma:

| Entidade | Tabela | Schema |
|----------|--------|--------|
| Turma (temporada) | `cohorts` | `(id, name, ...)` |
| Aula (sessão) | `classes` | `(id, weekday, cohort_id [DEPRECATED], ...)` |
| Join M:N | `class_cohorts` | `(class_id, cohort_id, weekday)` — introduzida 2026-05-16 |

**Problema:** `class_cohorts` criada tardiamente (20260516). Antes, `classes.cohort_id` era source-of-truth. Queries antigas continuam usando `classes.cohort_id` — não migradas pra `class_cohorts` JOIN.

**Riscos atuais em prod:**
- `get_weekly_attendance(p_cohort_id)` pode buscar via path inconsistente
- Multi-day classes (1 aula em 2 cohorts) só aparece via `class_cohorts` — queries antigas perdem
- Refactor futuro pode acidentalmente reintroduzir uso de coluna deprecated

**Memory check** (`multi-day-classes-schema`): `class_mentors.weekday` é fonte real do calendário; `classes.weekday` é só hint UI single-int. Confirma que mover pra `class_cohorts` é direção correta.

**Sistema EM PRODUÇÃO** — queries em runtime usam ambos paths. Drop sem migration completa = quebra silenciosa.

## Acceptance Criteria

### AC1 — Zero grep matches `classes.cohort_id` em código
**Given** codebase post-refactor
**When** `grep -rn "classes\.cohort_id" supabase/functions/ supabase/migrations/ admin/ --exclude-dir=node_modules`
**Then** retorna 0 matches (exceto migration que dropa a coluna)
**E** todas queries migram pra `class_cohorts` JOIN

### AC2 — RPCs críticas migradas
**Given** RPCs identificadas como usando `classes.cohort_id`
**When** refactor aplicado
**Then** `get_weekly_attendance(p_cohort_id)` usa JOIN class_cohorts
**E** `get_present_students(p_class_id)` usa JOIN class_cohorts
**E** Outras RPCs (mapeadas em T1) migradas

### AC3 — Backfill verificável: zero órfãos
**Given** 100% rows em `classes` que tinham `cohort_id`
**When** validation query
**Then** `SELECT count(*) FROM classes WHERE cohort_id IS NOT NULL AND NOT EXISTS (SELECT 1 FROM class_cohorts WHERE class_id = classes.id)` retorna 0
**E** backfill RPC `backfill_class_cohorts_from_legacy()` idempotente
**E** Se órfãos > 0 antes drop → BLOCK + alert admin

### AC4 — DEPRECATED comment 1 sprint antes drop
**Given** apply migration ADD COMMENT
**When** migration intermediária aplica
**Then** `COMMENT ON COLUMN classes.cohort_id IS 'DEPRECATED — use class_cohorts JOIN. Drop scheduled 2026-06-05.'`
**E** Slack alert agendado 7d antes drop
**E** Time admin notificado via Slack/email

### AC5 — Drop migration tem rollback
**Given** drop migration aplicado
**When** rollback necessário
**Then** migration down recria coluna `cohort_id text` + backfill via class_cohorts (primeira FK match)
**E** rollback testado local antes prod

### AC6 — Smoke test queries críticas
**Given** apply migration drop
**When** roda smoke
**Then** `SELECT * FROM get_weekly_attendance('T5')` retorna rows
**E** `SELECT * FROM get_present_students('<class_id>')` retorna rows
**E** `SELECT * FROM v_class_calendar` (se existe) retorna rows
**E** Zero erros "column cohort_id does not exist"

### AC7 — Audit log entry
**Given** migration aplica
**When** completa
**Then** `audit_log` event_type='dual_identity_cleanup' com payload (classes_migrated, orphans_resolved, rpcs_refactored)

### Template refactor RPC

```sql
-- BEFORE
CREATE OR REPLACE FUNCTION get_weekly_attendance(p_cohort_id text)
RETURNS TABLE(...)
AS $$
  SELECT ... FROM classes c WHERE c.cohort_id = p_cohort_id ...
$$;

-- AFTER
CREATE OR REPLACE FUNCTION get_weekly_attendance(p_cohort_id text)
RETURNS TABLE(...)
AS $$
  SELECT ...
  FROM classes c
  JOIN class_cohorts cc ON cc.class_id = c.id
  WHERE cc.cohort_id = p_cohort_id
$$;
```
