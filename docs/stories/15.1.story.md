## O que fazer

Implementar role `cs` no Supabase Auth com redirect role-based em `/admin/login`: usuários com `user_metadata.role='cs'` são redirecionados para `/cs/`, role='admin' para `/admin/`. Admin tem super-acesso a `/cs/*` (backward compat).

Esta story destrava: 15.5 (área CS), 15.2-15.7 (UIs CS), 15.D/15.F (integrações).

## Business Value

- **Pré-requisito de segurança:** sem role separado, CS team opera com privilégios admin (risco LGPD + escopo)
- **Reuso máximo Supabase Auth:** zero invenção — apenas estende user_metadata
- **3 ACs-1/2/3 do épico atendidos** (login flow definido em FR-1)
- **Habilita Wave 4 inteira** (UIs CS dependem de auth guard)

## Dev Notes

### Refactor verifyAdmin → verifyAdminOrCs

```typescript
// supabase/functions/_shared/auth.ts
export async function verifyAdminOrCs(authHeader: string | null): Promise<{ role: string; userId: string } | null> {
  if (!authHeader?.startsWith("Bearer ")) return null;
  const token = authHeader.slice(7);

  // Service-role bypass
  if (decodeJwtRole(token) === "service_role") return { role: "service_role", userId: "system" };

  const client = createClient(SUPABASE_URL, SUPABASE_ANON_KEY, { auth: { autoRefreshToken: false, persistSession: false } });
  const { data: { user } } = await client.auth.getUser(token);
  const role = user?.user_metadata?.role;

  if (role === "admin" || role === "cs") {
    return { role, userId: user!.id };
  }
  return null;
}
```

### Frontend Redirect Pattern (js/admin/auth.js)

```javascript
async function handleLogin(email, password) {
  const { data, error } = await supabase.auth.signInWithPassword({ email, password });
  if (error) return showError(error.message);

  const role = data.user.user_metadata?.role;

  switch (role) {
    case 'admin':
      window.location.href = '/admin/';
      break;
    case 'cs':
      window.location.href = '/cs/';
      break;
    default:
      await supabase.auth.signOut();
      showError('Acesso não autorizado para este usuário.');
  }
}
```
