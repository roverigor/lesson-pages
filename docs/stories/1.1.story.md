## Descrição

Como **coordenador da Academia Lendária**, preciso que as tabelas de suporte ao sistema de notificações WhatsApp existam no banco de dados Supabase com as RLS policies corretas, para que o fluxo de notificações event-driven possa ser implementado nas stories subsequentes sem risco à estrutura de dados existente.

**Contexto:** A arquitetura do sistema de notificações foi completamente documentada em `docs/architecture-notifications.md` (Aria, 2026-03-30). O schema SQL (`db/notifications-schema.sql`) já foi criado pelo arquiteto e aguarda execução. Esta story valida e aplica esse schema em produção.

---

## Acceptance Criteria

### AC1 — Tabela `mentors` criada
- **Dado** que o Supabase do projeto `gpufcipkajppykmnmdeh` está acessível
- **Quando** o schema SQL for executado
- **Então** a tabela `mentors` deve existir com: `id` (UUID PK), `name` (TEXT NOT NULL), `phone` (TEXT UNIQUE NOT NULL), `role` (TEXT CHECK em 'Professor'/'Host'/'Both'), `active` (BOOLEAN DEFAULT true), `created_at`, `updated_at`

### AC2 — Tabelas bridge N:N criadas
- **Dado** que `classes` e `cohorts` já existem
- **Quando** o schema for executado
- **Então** `class_cohorts` (class_id FK, cohort_id FK) e `class_mentors` (class_id FK, mentor_id FK, role TEXT) devem existir com constraints e índices apropriados

### AC3 — Tabela `notifications` criada com todos os campos
- **Dado** o schema em `docs/architecture-notifications.md §3.1`
- **Quando** o schema for executado
- **Então** a tabela `notifications` deve ter todos os campos especificados, incluindo: `status` TEXT com default 'pending', `retry_count` INT DEFAULT 0, `max_retries` INT DEFAULT 3, `evolution_response` JSONB, `metadata` JSONB

### AC4 — RLS Policies aplicadas corretamente
- **Dado** que o projeto usa autenticação Supabase com roles via JWT metadata
- **Quando** as políticas forem aplicadas
- **Então:**
  - `notifications`: SELECT, INSERT, UPDATE restrito a `admin` (role via JWT)
  - `mentors`: SELECT para qualquer authenticated; INSERT/UPDATE/DELETE para `admin` only
  - `class_cohorts` e `class_mentors`: SELECT para authenticated; escrita para `admin` only

### AC5 — Dados seed de mentores populados
- **Dado** que há mentores ativos no sistema (Talles, Klaus, Day, Bruno Gentil, etc.)
- **Quando** o seed for executado
- **Então** a tabela `mentors` deve ter ao menos os mentores ativos com phones no formato internacional (55DD9XXXXXXXX)

### AC6 — Dados de relacionamento populados
- **Dado** que `classes` e `cohorts` existem com dados reais
- **Quando** o seed for executado
- **Então** `class_cohorts` deve ter mapeamento correto de quais classes atendem quais cohorts; `class_mentors` deve ter quais mentores ensinam quais classes com seus papéis

### AC7 — Zero impacto nas tabelas existentes
- **Dado** que `classes`, `cohorts`, `attendance`, `students` estão em produção com dados reais
- **Quando** a migration for executada
- **Então** nenhuma dessas tabelas deve ser alterada, e todos os dados existentes permanecem intactos

### AC8 — Verificação pós-migração
- **Dado** que o schema foi aplicado
- **Quando** a verificação for feita via Supabase Dashboard
- **Então** todas as tabelas aparecem na lista, índices existem, e um INSERT de teste em `notifications` (status='pending') é aceito e pode ser deletado sem erros

### Padrão de RLS do projeto
O projeto usa a seguinte verificação de admin role:
```sql
(auth.jwt() ->> 'user_metadata')::jsonb ->> 'role' = 'admin'
```
Verificar no `db/schema.sql` existente para manter consistência exata.

### Formato de phone para Evolution API
Phones devem ser armazenados sem `+` e sem espaços: `5511999999999`
