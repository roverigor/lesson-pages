## Story
Como **administrador da Academia Lendária**,
Eu preciso que **tabela `notifications` (delivery_status) tenha coluna `provider` (`evolution`|`meta`) NOT NULL + backfill heurístico + refactor `dispatch-retry` pra rotear via coluna**,
Para que **retries sempre cheguem ao provider correto e dispatchers gravem provider em todo INSERT (sem heurística runtime)**.

## Contexto

Discovery `01-architecture-map.md` §4.4 identificou:

- 2 webhooks delivery: `delivery-webhook` (Evolution) + `meta-delivery-webhook` (Meta)
- Ambos escrevem em `notifications` table (delivery_status columns)
- **Sem coluna `provider`** → `dispatch-retry` infere via heurística (ID format `wamid.*` = Meta, else Evolution)
- Heurística falha silenciosamente se ID format mudar

**Tabela alvo:** Discovery cita "delivery_status table" mas implementação atual stora em `notifications` (conforme `20260402200000_delivery_status.sql`). Migration adiciona em `notifications` (confirmar com @data-engineer pré-apply se separada).

**Dispatchers atuais que INSERT em notifications:**
- dispatch-survey
- dispatch-class-nps
- dispatch-class-reminders
- dispatch-ps-rsvp
- admin-send-group-once

Todos precisam atualizar pra gravar `provider` no INSERT.

## Acceptance Criteria

### AC1 — Coluna provider NOT NULL pós-backfill
**Given** schema atual sem coluna provider
**When** migration 2-step aplica
**Then** Step 1: ADD COLUMN provider text DEFAULT 'evolution' + CHECK constraint `IN ('evolution', 'meta')`
**E** Step 2 (após backfill): ALTER COLUMN provider SET NOT NULL
**E** índice `idx_notifications_provider`

### AC2 — Backfill heurística
**Given** rows existentes
**When** backfill RPC executa
**Then** WHERE `evolution_message_ids IS NOT NULL` → provider='evolution'
**E** WHERE `meta_message_id IS NOT NULL` OR `external_id LIKE 'wamid.%'` → provider='meta'
**E** WHERE ambíguo (ID null ou format inesperado) → Slack alert pra review humano + provider DEFAULT 'evolution' (mais comum legacy)
**E** backfill idempotente

### AC3 — Zero NULL antes NOT NULL
**Given** AC2 backfill executou
**When** validation query
**Then** `SELECT COUNT(*) FROM notifications WHERE provider IS NULL` = 0
**E** gate: se > 0 → BLOCK Step 2 NOT NULL apply

### AC4 — dispatch-retry refactor (coluna-driven)
**Given** dispatch-retry edge function
**When** processa row pra retry
**Then** lê `notification.provider` diretamente
**E** roteia: `if provider == 'evolution': call Evolution API` / `if provider == 'meta': call Meta API`
**E** ZERO heurística baseada em ID format
**E** audit em `dispatch_retry_audit` (criar se não existe)

### AC5 — Test retry triplo
**Given** local PG 15 ou staging
**When** dispara retry pra:
- 1 message com provider='evolution' (mock Evolution API call)
- 1 message com provider='meta' (mock Meta API call)
**Then** ambos roteiam pra API correta
**E** logs estruturados mostram routing decision
**E** audit entries criadas

### AC6 — Slack alert se backfill ambíguo
**Given** AC2 backfill encontra rows com ID format inesperado
**When** alert dispara
**Then** Slack #devops recebe count + 5 sample external_id
**E** admin pode revisar manualmente + UPDATE manual se necessário

### AC7 — Todos dispatchers gravam provider
**Given** grep `INSERT INTO notifications` em todos dispatchers
**When** refactor aplica
**Then** cada INSERT contém `provider` column explicit
**E** validation: `grep -rn "INSERT INTO.*notifications" supabase/functions/ | wc -l` matches count de "provider":
```
dispatch-survey: provider='meta' default (depends destino)
dispatch-class-nps: provider mixed (Meta DM + Evolution group)
dispatch-class-reminders: provider='evolution'
dispatch-ps-rsvp: provider='meta'
admin-send-group-once: provider='evolution'
```

### AC8 — Idempotência migration
**Given** migration rerun
**When** segunda execução
**Then** ADD COLUMN IF NOT EXISTS não falha
**E** CHECK constraint reusável
**E** índice IF NOT EXISTS

## Dev Notes

### Stack

- Edge functions: 5 dispatchers + dispatch-retry
- Pattern UPSERT/INSERT existente em cada dispatcher

### Template backfill RPC

```sql
CREATE OR REPLACE FUNCTION public.backfill_notifications_provider()
RETURNS TABLE(provider_inferred text, rows_updated bigint, ambiguous_count bigint)
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_evolution bigint;
  v_meta bigint;
  v_ambiguous bigint;
BEGIN
  -- Evolution: evolution_message_ids populated
  UPDATE notifications
     SET provider = 'evolution'
   WHERE provider IS NULL
     AND evolution_message_ids IS NOT NULL;
  GET DIAGNOSTICS v_evolution = ROW_COUNT;

  -- Meta: external_id começa com 'wamid.' OR meta_message_id populated
  UPDATE notifications
     SET provider = 'meta'
   WHERE provider IS NULL
     AND (external_id LIKE 'wamid.%' OR meta_message_id IS NOT NULL);
  GET DIAGNOSTICS v_meta = ROW_COUNT;

  -- Ambíguo: log + default evolution
  SELECT count(*) INTO v_ambiguous
    FROM notifications
   WHERE provider IS NULL;

  IF v_ambiguous > 0 THEN
    INSERT INTO audit_log (event_type, payload)
    VALUES (
      'notifications_provider_backfill_ambiguous',
      jsonb_build_object('count', v_ambiguous, 'sample', (SELECT array_agg(external_id) FROM (SELECT external_id FROM notifications WHERE provider IS NULL LIMIT 5) s))
    );
    -- Default fallback
    UPDATE notifications SET provider = 'evolution' WHERE provider IS NULL;
  END IF;

  RETURN QUERY VALUES
    ('evolution', v_evolution, 0::bigint),
    ('meta', v_meta, 0::bigint),
    ('ambiguous_default_evolution', v_ambiguous, v_ambiguous);
END;
$$;
```
