## Story
Como **administrador da Academia Lendária**,
Eu preciso que **3 tabelas suspeitas órfãs (`pending_student_assignments`, `notification_queue`, `alert_history`) sejam auditadas, decididas (keep/deprecate/drop), e o schema.md seja atualizado**,
Para que **diagnósticos futuros não percam tempo investigando tabelas mortas**.

## Contexto

Discovery `02-db-audit.md` §5 listou 3 tabelas suspeitas:

| Tabela | Última escrita | Status investigação |
|--------|----------------|---------------------|
| `pending_student_assignments` | ? | Nunca mencionada em migrations recentes — ÓRFÃ? |
| `notification_queue` | 20260417110000 criação | Nenhuma query visível — ÓRFÃ? |
| `alert_history` | ? | Não aparece em migrations recentes — ÓRFÃ? |

Investigação real exige:
- Audit 90 dias logs Supabase (writes + reads)
- Grep codebase pra references
- Cross-ref com `find_duplicate_students()`, dispatch flows, RPCs

**Risco baixo:** drop sem deprecation = reintroduzir tabela é custo médio. 1 sprint deprecation buffer.

## Acceptance Criteria

### AC1 — Decisão documentada por tabela
**Given** 3 tabelas suspeitas
**When** investigação completa
**Then** cada uma tem decisão em ADR-026:
- `pending_student_assignments`: keep | deprecate | drop
- `notification_queue`: keep | deprecate | drop
- `alert_history`: keep | deprecate | drop
**E** razão documentada (logs evidence + grep results)

### AC2 — Se Keep: comment SQL justificativa
**Given** decisão "keep"
**When** migration aplica
**Then** `COMMENT ON TABLE <tbl> IS 'Active. Used by <consumer>. Ref ADR-026.'`

### AC3 — Se Deprecate: rename _deprecated_YYYYMMDD + alert
**Given** decisão "deprecate"
**When** migration aplica
**Then** `ALTER TABLE <tbl> RENAME TO <tbl>_deprecated_20260522`
**E** trigger AFTER INSERT em renamed table: INSERT audit_log + Slack alert (write inesperado pós-deprecate)
**E** 1 sprint depois: drop migration follow-up

### AC4 — Se Drop: backup snapshot antes
**Given** decisão "drop"
**When** migration aplica
**Then** `CREATE TABLE <tbl>_snapshot_20260522 AS SELECT * FROM <tbl>` (cópia completa)
**E** snapshot table sem RLS (admin-only read via service_role)
**E** drop migration após snapshot OK
**E** Rollback: rename snapshot back

### AC5 — schema.md atualizado
**Given** decisões aplicadas
**When** PR mergeado
**Then** `docs/architecture/schema.md` (se existir, senão criar) contém:
- Lista tabelas ativas (com purpose)
- Lista tabelas deprecated (com data + razão)
- Lista tabelas dropadas (com snapshot reference)

### AC6 — ADR-026 housekeeping completo
**Given** decisões + evidência
**When** ADR criado
**Then** `docs/architecture/ADR-026-orphan-tables-cleanup.md` contém:
- Investigação methodology (audit logs + grep)
- Evidence por tabela
- Decisão + rationale
- Rollback procedure se drop
