## Story
Como **administrador da Academia Lendária**,
Eu preciso que **dispatch NPS pós-aula use um único motor consolidado (`dispatch-survey`) em vez de 3 caminhos paralelos (`dispatch-class-nps` + `dispatch-survey` + `send-whatsapp` legacy)**,
Para que **manutenção fique 3x mais barata, dashboards sejam coerentes (1 schema vs 3), e novas features de dispatch sejam buildadas em cima de fundação única**.

## Contexto

Discovery `01-architecture-map.md` §4.1 identificou **3 caminhos paralelos** de dispatch NPS em produção:

| Path | Edge function | Schema/tabela | Trigger | Status |
|------|--------------|---------------|---------|--------|
| **A** | `dispatch-class-nps` | `nps_class_dispatch_jobs` + `nps_class_links` | cron 5m + manual | Ativo — EPIC-021 Hub |
| **B** | `dispatch-survey` | `surveys` + `survey_responses` + `dispatch_history` | admin one-shot | Ativo (mais novo, throttle 500ms) |
| **C** | `send-whatsapp` | `notifications` table + delivery_status | DB webhook (legacy) | Legacy (2026-04-22) |

**Problema raiz:**
- 3 schemas distintos (`nps_*`, `survey_*`, `notifications`) → 3 lugares pra auditar entregas
- 3 dashboards (`nps-results`, `nps-monitor`, `envios`) consultam fontes diferentes
- Manutenção 3x: bug em template Meta exige fix em 3 lugares
- Path C (send-whatsapp) é legacy DB webhook — fluxo opaco, sem observability moderna

**Por que dispatch-survey canonical:**
- Mais novo (2026-05-22 mod, vs 2026-05-20 path A vs 2026-04-22 path C)
- Throttle 500ms entre sends (anti-rate-limit)
- Hybrid Meta+Evolution (não força um canal)
- Schema generic — aceita qualquer tipo survey (nps_class, ps_rsvp, etc)
- `dispatch_history` já mais rica (dispatch_id, form_id, session_index, etc)

**Decisão arquitetural:**
- `dispatch-survey` é target canonical → aceita tipo `nps_class` mapeando pra surveys generic
- Feature flag `nps_dispatch_engine` em `app_config`: `legacy` (default) | `unified`
- Flip atômico via admin RPC (não env var → muda sem redeploy)
- Rollback bidirecional (revert flag → engine legacy continua OK)
- `nps_class_dispatch_jobs` lock idempotency preservado (não duplicar trigger)
- `send-whatsapp` marcado dormant + comentário em código + trigger notifications desativado 30d post-flip

**Sistema EM PRODUÇÃO** com 1k+ alunos ativos + envios diários — flip exige smoke em turma sentinela + Slack alert + gate humano + janela manutenção.

## Acceptance Criteria

### AC1 — dispatch-survey aceita tipo `nps_class`
**Given** request `POST /functions/v1/dispatch-survey` com body `{ "survey_type": "nps_class", "class_id": "...", "cohort_id": "..." }`
**When** flag `nps_dispatch_engine = 'unified'`
**Then** edge function gera entries equivalentes em `surveys` + `dispatch_history` + `survey_links` (espelho semântico de `nps_class_links`)
**E** preserve `nps_class_dispatch_jobs` lock idempotency (mesmo class_id não dispara 2x)
**E** Meta DM + Evolution group ambos disparam (hybrid existente preservado)

### AC2 — Feature flag `nps_dispatch_engine`
**Given** tabela `app_config`
**When** migration aplica
**Then** existe row `key='nps_dispatch_engine', value='legacy'` (default seguro)
**E** RPC `set_nps_dispatch_engine(text)` aceita `'legacy'` | `'unified'`
**E** RPC valida value (rejeita outros valores)
**E** RPC requer `is_dashboard_admin()` (não authenticated comum)

### AC3 — Flip atômico via admin RPC + Slack
**Given** flag value muda via RPC
**When** RPC executa
**Then** UPDATE `app_config` em transação BEGIN/COMMIT
**E** trigger INSERT em `audit_log` event_type='nps_engine_flip' com old_value, new_value, operator
**E** Slack #devops alert disparado (curl direto na RPC ou via wrapper)
**E** flag valor novo aplica IMEDIATAMENTE no próximo dispatch (sem redeploy)

### AC4 — Rollback bidirecional
**Given** flag flipped pra `unified` e bug detectado
**When** admin chama `set_nps_dispatch_engine('legacy')`
**Then** dispatch-class-nps (Path A) retoma controle
**E** envios subsequentes usam schema legacy
**E** entries já em `dispatch_history` (unified) ficam preservados (audit trail)
**E** Slack alert disparado em ambas direções (flip + rollback)

### AC5 — send-whatsapp marcado dormant
**Given** Path C (send-whatsapp + notifications) está ativo
**When** migration aplica
**Then** trigger DB que escreve em `notifications` é DESATIVADO (DROP TRIGGER ou DISABLE)
**E** edge function `send-whatsapp` recebe comentário no topo: `// DORMANT — replaced by dispatch-survey post-2026-05-22. Remove after 2026-06-22.`
**E** rota Supabase function permanece deployada 30d (rollback ultimate)
**E** `notifications` table mantida (audit)

### AC6 — Migration schema `dispatch_type` enum
**Given** `dispatch_history` table
**When** migration aplica
**Then** coluna `dispatch_type text` adicionada com CHECK constraint `dispatch_type IN ('nps_class', 'ps_rsvp', 'survey_generic', 'reminder')`
**E** rows existentes recebem `dispatch_type = 'survey_generic'` (default seguro)
**E** índice `idx_dispatch_history_type` criado

### AC7 — Smoke turma sentinela (pre-flip)
**Given** flag ainda `legacy`
**When** admin dispara dispatch-survey com `survey_type='nps_class'` + `class_id` da turma sentinela
**Then** entries geradas em `surveys` + `dispatch_history` (não em `nps_class_links`)
**E** 1 student da turma sentinela recebe Meta DM
**E** comparação manual side-by-side: legacy path (existing) vs unified path (new) producem resultados equivalentes
**E** smoke documentado em `qa/22.2-sentinel-smoke.md`

### AC8 — Slack alert no flip
**Given** AC3 implementado
**When** RPC flip executa
**Then** Slack #devops recebe blocks:
- Header: "NPS engine FLIP: {old} → {new}"
- Operator, timestamp, audit_log_id
- Quick rollback instruction (RPC call pra reverter)

### AC9 — Audit log entry
**Given** `audit_log` recebe inserts dos eventos
**When** flip ou rollback executa
**Then** event_type='nps_engine_flip' com payload:
- old_value, new_value, operator, dispatched_at
- diff: surveys_created, dispatch_history_rows (snapshot pré-flip)
- next_action: "monitor dispatch-survey logs pra anomalias"

### AC10 — Documentação runbook
**Given** complexidade do rollback
**When** PR mergeado
**Then** `docs/runbooks/22.2-nps-engine-flip.md` existe contendo:
- Pré-checklist (smoke sentinela OK + Slack on-call + janela)
- Comando flip (RPC call)
- Sinais de problema (dispatch-survey logs com error rate >5%, Slack alerts cascata)
- Rollback step-by-step (RPC reverter + check)
- Post-mortem template

### Template RPC set_nps_dispatch_engine

```sql
CREATE OR REPLACE FUNCTION public.set_nps_dispatch_engine(p_value text)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_old_value text;
  v_operator  text;
BEGIN
  -- Gate
  IF NOT public.is_dashboard_admin() THEN
    RAISE EXCEPTION 'Permission denied: admin only';
  END IF;

  -- Validate
  IF p_value NOT IN ('legacy', 'unified') THEN
    RAISE EXCEPTION 'Invalid engine value: %. Allowed: legacy, unified', p_value;
  END IF;

  -- Snapshot
  SELECT value INTO v_old_value FROM public.app_config WHERE key = 'nps_dispatch_engine';
  v_operator := COALESCE(auth.jwt()->'user_metadata'->>'email', 'unknown');

  -- Flip
  UPDATE public.app_config
     SET value = p_value, updated_at = now()
   WHERE key = 'nps_dispatch_engine';

  -- Audit
  INSERT INTO public.audit_log (event_type, payload)
  VALUES (
    'nps_engine_flip',
    jsonb_build_object(
      'old_value', v_old_value,
      'new_value', p_value,
      'operator', v_operator,
      'flipped_at', now(),
      'story_id', '22.2'
    )
  );

  RETURN jsonb_build_object('ok', true, 'old', v_old_value, 'new', p_value);
END;
$$;

GRANT EXECUTE ON FUNCTION public.set_nps_dispatch_engine(text) TO authenticated;
```
