## Story
Como **administrador da Academia Lendária**,
Eu preciso que **TODAS tabelas com dados sensíveis tenham Row Level Security (RLS) habilitado + POLICY restritiva (não permissiva), coerente com o Tier de sensibilidade**,
Para que **nenhum usuário autenticado consiga ler dados de outros (ex: respostas NPS, attendance, student_imports, magic-link tokens) e o sistema fique aderente ao baseline de segurança Supabase**.

## Contexto

Discovery `02-db-audit.md` §6 identificou 39 tabelas sem RLS. Audit @data-engineer (`22.4.data-engineer-notes.md` Q1) descobriu realidade mais nuançada: **6 das 7 tabelas Tier 1 já têm RLS habilitado E POLICY, mas as policies são frouxas** (`USING (true)`, `auth.uid() IS NOT NULL`, inline role checks distintos). Não é "gap fix" puro — é **hardening de policies permissivas existentes + ENABLE em tabelas faltantes**.

**Realidade pré-migration (audit 2026-05-22):**
- **Tier 1 (7 tabelas):** 6/7 já com RLS+POLICY frouxa (replace obrigatório) + `staff` sem RLS (create)
- **Tier 2 (16 tabelas):** mix — alguns já com POLICY OK (preserve), outros sem POLICY (create), 3 sem ENABLE
- **Tier 3 (4 tabelas):** sem POLICY (create) — incluindo `app_config` que requer refactor pg_cron antes (T11)
- **Exceções (4 tabelas):** `ac_dispatch_callbacks`, `oauth_states`, `ac_purchase_events`, `ac_product_mappings` — mantém sem POLICY (service_role only), justificadas em ADR-020

**Vazamentos ATIVOS em prod (audit @data-engineer):**
- `class_nps_responses` POLICY `nps_responses: read for auth` USING (true) → qualquer authenticated lê todas respostas + comentários livres
- `wa_group_members` POLICY `Authenticated full access` USING (auth.uid() IS NOT NULL) → qualquer authenticated lê phones de toda base
- `nps_class_links` POLICY `nps_links: read for auth` USING (true) → tokens magic-link expostos (risco spoofing de respostas)
- `student_attendance` POLICY authenticated read all → presença individual vazada

**Decisão arquitetural confirmada (data-engineer Q1):** Não existe link `students.id` ↔ `auth.users.id` no schema. Painel é admin-only — alunos não logam no Supabase Auth. Self-read via `student_id = auth.uid()` é semanticamente impossível pra todas Tier 1. **Pattern Tier 1 default: `is_dashboard_admin()` only** (Opção B do data-engineer Q3) + comment SQL `-- Future: adicionar OR <ownership_clause> when self-service NPS view ships` pra extensão futura.

**Discrepância 39 vs 27:** discovery contou versões intermediárias em migrations / tabelas dropadas. Audit reconciliado = 27 escopo + 4 exceções. AC #1 alvo é a lista nominal do Anexo Epic reconciliado, com recontagem final em prod via `pg_class.relrowsecurity` antes de fechar a story.

**Sistema EM PRODUÇÃO** com 1k+ alunos ativos e envios diários — migration obriga idempotência + dry-run + rollback + smoke + Slack alert + refactor pg_cron testado antes de RLS apply em `app_config`.

### AC4 — Migration idempotente (REPLACE pattern PG 15)
**Given** migration aplicada uma vez

**Then** não falha, não duplica POLICY, retorna mesmo estado
**E** usa pattern obrigatório em TODOS statements de POLICY (PG 15 compat — `CREATE POLICY IF NOT EXISTS` NÃO existe até PG 16):
```sql
DROP POLICY IF EXISTS "<policy_name>" ON public.<table>;
CREATE POLICY "<policy_name>" ON public.<table>
  FOR <cmd> TO <role> USING (<qual>) [WITH CHECK (<qual>)];
```
**E** usa `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` (idempotente em Postgres ≥9.5)
**E** rerun completo (apply → apply) não gera diff em `pg_policies` (validar via dump comparativo)

#### Tier 1 — `is_dashboard_admin()` only + future-extension comment

```sql
-- TEMPLATE TIER 1 — admin lê via is_dashboard_admin(), service_role bypass natural
-- Future: adicionar OR <ownership_clause> when self-service NPS view ships
BEGIN;

ALTER TABLE public.<table> ENABLE ROW LEVEL SECURITY;

-- DROP policies antigas com IF EXISTS pra idempotência cross-rerun
DROP POLICY IF EXISTS "<old_policy_name>" ON public.<table>;  -- existing frouxa
DROP POLICY IF EXISTS "<table>_select_admin" ON public.<table>;
DROP POLICY IF EXISTS "<table>_write_admin" ON public.<table>;

CREATE POLICY "<table>_select_admin" ON public.<table>
  FOR SELECT TO authenticated
  USING (public.is_dashboard_admin());
  -- Future: adicionar OR student_id IN (SELECT ... ) when self-service NPS view ships

CREATE POLICY "<table>_write_admin" ON public.<table>
  FOR ALL TO authenticated
  USING (public.is_dashboard_admin())
  WITH CHECK (public.is_dashboard_admin());

COMMIT;
```

### Cuidados críticos

1. **USING vs WITH CHECK** — não confundir:
   - `USING`: filtro em SELECT/UPDATE (linhas visíveis)
   - `WITH CHECK`: validação em INSERT/UPDATE (linhas permitidas escrever)
   - Para Tier 1 write admin: precisa AMBOS

2. **Tier 1 sem coluna ownership clara (todas as 7):**
   - @data-engineer Q1 confirmou: **NÃO existe link `students.id` ↔ `auth.users.id`** em nenhuma migration.
   - Self-read via `student_id = auth.uid()` é semanticamente impossível.
   - **Pattern default: `is_dashboard_admin()` only** (Opção B confirmada).
   - Comment SQL `-- Future: adicionar OR <ownership_clause> when self-service NPS view ships` documenta extensão futura sem mudar arquitetura agora.

3. **Service role bypass:**
   - Edge functions DEVEM usar `SUPABASE_SERVICE_ROLE_KEY` (não anon key)
   - Confirmar via grep antes de apply: `grep -r "SUPABASE_ANON_KEY" supabase/functions/` — flag suspeitos

4. **Função `is_dashboard_admin()`:**
   - Versão única em `20260516020300_helper_functions.sql:25-30`
   - Body: `SELECT COALESCE((auth.jwt()->'user_metadata'->>'role') = 'admin', false)` — STABLE
   - GRANT EXECUTE pra `authenticated`, `anon`, `service_role`
   - **Incluir `CREATE OR REPLACE` no header da migration** (defensivo — AC10)
   - **NÃO autoriza role `cs`** — vide caveat `response_metadata` em T1

5. **2-step se necessário** (apply RLS ENABLE separado de POLICY) — preferir 1-step idempotente via BEGIN/COMMIT por tabela
   - Risco real: app pode quebrar entre ENABLE e CREATE POLICY (windowzinha de 401)
   - Mitigação: transação BEGIN ... COMMIT englobando ambos por tabela

6. **Pg_cron refactor (T11) PRECEDE RLS apply em app_config:**
   - Sequência obrigatória: T11 refactor pg_cron → smoke pré-RLS → aplicar Tier 3 template em app_config → smoke pós-RLS
   - Risco se invertido: pg_cron quebra em prod entre RLS apply e job próximo trigger

7. **Verificar policies existentes:**
   ```sql
   SELECT tablename, policyname, cmd, qual FROM pg_policies
   WHERE schemaname='public' AND tablename = ANY(ARRAY[/* 27 */])
   ORDER BY tablename;
   ```
   - Se já existir POLICY parcial — não duplicar, completar via `DROP IF EXISTS` + `CREATE`

8. **`response_metadata` + role `cs`:**
   - Hoje usa inline `(auth.jwt()->'user_metadata'->>'role') IN ('admin','cs')`
   - Padronizar pra `is_dashboard_admin()` quebra acesso CS se houver user CS ativo
   - **T1 valida:** `SELECT raw_user_meta_data->>'role' AS r, count(*) FROM auth.users GROUP BY 1`
   - Se CS ativo → manter inline check em `response_metadata` apenas (decisão local) OU expandir `is_dashboard_admin()` (flag pra @architect — fora escopo S.022.4)
