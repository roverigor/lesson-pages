## Descrição

Como **coordenador da Academia Lendária**, preciso que o sistema envie lembretes de aula automaticamente no horário configurado — sem intervenção manual — para que os mentores e grupos recebam avisos consistentes mesmo quando não estou disponível para enviar manualmente.

**Contexto:** EPIC-001 entregou o sistema de notificações manual. Esta story fecha o gap de automação: cria a tabela de agendamentos, habilita pg_cron no Supabase, e implementa a função que gera notificações automaticamente com base nos schedules configurados.

## Acceptance Criteria

### AC1 — Extensão pg_cron habilitada
- **Dado** que o projeto Supabase `gpufcipkajppykmnmdeh` está acessível
- **Quando** a migration for aplicada
- **Então** `SELECT * FROM pg_extension WHERE extname = 'pg_cron'` deve retornar 1 linha

### AC2 — Tabela `notification_schedules` criada
- **Quando** a migration for aplicada
- **Então** a tabela deve existir com: `id`, `class_id` (FK→classes), `cohort_id` (FK→cohorts), `notification_type`, `target_type`, `message_template`, `hours_before` (DEFAULT 2), `active` (DEFAULT true), `last_fired_at`, `next_fire_at`, `created_by`, `created_at`, `updated_at`

### AC3 — RLS em `notification_schedules`
- **Dado** que a tabela usa dados sensíveis de configuração
- **Quando** as políticas forem aplicadas
- **Então** SELECT/INSERT/UPDATE/DELETE restrito a `admin` (mesmo padrão de `notifications`)

### AC4 — Função `process_notification_schedules()` implementada
- **Quando** a função for chamada
- **Então** para cada schedule `active = true` com `next_fire_at <= now()` e `next_fire_at > last_fired_at`:
  1. Verifica guard de duplicata (não inserir se já existe notification para mesma class nas últimas 2 horas)
  2. Faz INSERT em `notifications` com os campos do schedule e `status = 'pending'`
  3. Atualiza `last_fired_at = now()`
  4. Recalcula `next_fire_at` para a próxima ocorrência (próximo weekday da classe)

### AC5 — Job pg_cron registrado
- **Quando** a migration for aplicada
- **Então** `SELECT * FROM cron.job WHERE jobname = 'notify-schedules'` deve retornar 1 linha com schedule `*/15 * * * *`

### AC6 — Guard de duplicatas funcional
- **Dado** que um schedule foi disparado e gerou uma notification nas últimas 2 horas
- **Quando** a função for chamada novamente
- **Então** NÃO deve gerar nova notification para a mesma classe — idempotência garantida

### AC7 — Cálculo de `next_fire_at` correto
- **Dado** uma classe com `weekday = 3` (Quarta) e `time_start = '18:00'` e `hours_before = 2`
- **Quando** `next_fire_at` for calculado
- **Então** deve retornar a próxima Quarta-feira às 16:00 (BRT = UTC-3, verificar timezone)

### AC8 — Trigger `updated_at` em `notification_schedules`
- **Quando** um schedule for atualizado
- **Então** `updated_at` deve ser atualizado automaticamente

## Dev Notes

### Verificar disponibilidade de pg_cron
Antes de habilitar, confirmar:
```sql
SELECT * FROM pg_available_extensions WHERE name = 'pg_cron';
```
Se não disponível no plano atual, alternativa é usar Vercel Cron + chamada à Edge Function.

### Habilitar pg_cron
```sql
CREATE EXTENSION IF NOT EXISTS pg_cron;
GRANT USAGE ON SCHEMA cron TO postgres;
```

### Cálculo de next_fire_at
```sql
-- Próxima ocorrência de weekday X a partir de now()
SELECT
  date_trunc('day', now() AT TIME ZONE 'America/Sao_Paulo')
  + (((c.weekday - EXTRACT(DOW FROM now() AT TIME ZONE 'America/Sao_Paulo')::int + 7) % 7)
     || ' days')::interval
  + c.time_start::interval
  - (s.hours_before || ' hours')::interval AS next_fire
FROM classes c, notification_schedules s WHERE ...;
```

### Registro do job pg_cron
```sql
SELECT cron.schedule(
  'notify-schedules',
  '*/15 * * * *',
  'SELECT process_notification_schedules()'
);
```

### Guard de duplicatas
```sql
WHERE NOT EXISTS (
  SELECT 1 FROM notifications n
  WHERE n.class_id = s.class_id
    AND n.type = s.notification_type
    AND n.created_at > now() - interval '2 hours'
)
```
