## Contexto do incidente (27–28/05/2026)

NPS pós-aula não saiu completo. Dois bugs distintos identificados:

1. **T2 — job travado:** `dispatch-class-nps` bateu time budget 130s, abortou e deixou job `in_progress` sem `finished_at`. Cron `*/5min` pulou (vê `in_progress`) → 78 DMs nunca saíram.
2. **T5 — trigger pegou turma errada:** duas `classes` com mesmo `zoom_meeting_id` (uma `[DUP-DESATIVADA]` active=false sem cohorts, uma ativa). Trigger `trg_enqueue_nps_after_zoom_processed` fazia `SELECT ... WHERE zoom_meeting_id=X LIMIT 1` sem filtrar `active` nem ordenar → pegou desativada → 0 enqueues → 0 DMs.

Paliativos já aplicados:
- UPDATE class DUP-DESATIVADA zoom_meeting_id=NULL
- Migration `20260528120000_nps_trigger_active_class_guard.sql` com `active=true` + ORDER BY + warning
- Reenvio manual via Meta API (75 T5 + 78 T2)
- Edge fn `dispatch-class-nps` v18 deployed com graceful abort (status=pending no time budget)

Esta story = **correção estrutural** (classe do problema), não remendo.

## Fix 1 — UNIQUE parcial classe ativa por meeting

**Causa-raiz:** `zoom_meeting_id` não é único entre turmas. Filtro no trigger é band-aid. Garantia estrutural via banco.

```sql
CREATE UNIQUE INDEX CONCURRENTLY uq_classes_active_zoom_meeting
  ON classes (zoom_meeting_id)
  WHERE active = true AND zoom_meeting_id IS NOT NULL;
```

Pré-requisito: se Passo 0.4 achou conflito atual, listar pra decisão humana antes de criar índice.

Defesa em profundidade: mantém `active=true` no trigger.

## Fix 2 — Job não trava (unlock no abort + reaper)

Aplicar **consistente em todos dispatchers** do Passo 0.2.

### 2.1 Unlock no abort
Time budget hit + catch/finally → job volta `pending` + `last_progress_at = now()`. Retomada via links com status próprio (já idempotent).

### 2.2 Reaper (garantia real)
Início de cada execução do dispatcher, ANTES de pegar job:

```sql
UPDATE nps_class_dispatch_jobs
  SET status = 'pending'
  WHERE status = 'in_progress'
    AND COALESCE(last_progress_at, started_at) < now() - interval '10 minutes';
```

Razão: não dá pra confiar que função sempre rode cleanup — crash/timeout/deploy podem matar `finally`. Reaper = auto-cura.

### 2.3 RPC manual reclaim
`reclaim_stuck_dispatch_jobs()` chamável pela UI `/admin/nps-monitor` (crons dormentes — operador destrava sem SQL).

### 2.4 Idempotência (confirmar/garantir)
- Retomada envia só links `pending`
- Marca cada link `sent` imediatamente após 200 Meta (guarda message_id)
- Dedup phone (aluno em N cohorts não recebe N DMs)

Se já é assim, documentar. Senão, corrigir.

### 2.5 Coluna last_progress_at
Migration nova se ainda não existir.

## Fix 3 — Falha silenciosa → alertas

Observabilidade. Usar `send_slack_alert` + `recently_alerted` (throttle já existe).

### Alerta A — enqueue zero
Quando `meeting.ended` processado (ou `nps-preflight-check`), se turma resolvida esperava `> 0` alunos mas jobs/enqueues criados = 0 → Slack ("reunião X terminou, esperava ~N, criou 0 jobs"). T5 caso.

### Alerta B — job preso
Quando reaper recuperar job (ou check periódico) → Slack com job_id, turma, contagem pending/sent. T2 caso.
