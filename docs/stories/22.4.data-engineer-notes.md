## Resumo executivo (TL;DR)

1. **Não existe link `students.id` → `auth.users.id` no schema.** A coluna `id` em `students` é UUID local gerado por `gen_random_uuid()`. Não há `user_id`, `auth_id`, nem migration histórica que tenha adicionado FK pra `auth.users`. **Isso descarta o pattern "self read" (Opção A do prompt) pra todas as tabelas Tier 1.** Não existe self-service de aluno autenticado no painel (só admin login + magic-link por token).
2. **Conclusão Tier 1:** todas as 7 tabelas Tier 1 vão usar **Opção B — `is_dashboard_admin()` only** + bypass natural de `service_role`. Dev Notes da story 22.4 (linhas 282-286) chamam isso de "fallback `is_dashboard_admin() OR EXISTS(...)`" — promovo a padrão default.
3. **`is_dashboard_admin()` tem 1 versão única em prod** (migration `20260516020300_helper_functions.sql`). Body: `(auth.jwt()->'user_metadata'->>'role') = 'admin'`. Idêntica ao pattern inline usado por baseline (`cohorts`, `students`, `classes`). Recomendação: **incluir `CREATE OR REPLACE` na migration S.022.4 (defensivo)**, custo zero, evita race se função for dropada acidentalmente.
4. **Anexo S.022.4 do epic está parcialmente desatualizado.** Várias tabelas listadas como "RLS enabled sem policy" JÁ tem POLICY no arquivo de criação. Detalhes na seção Q1 abaixo + recomendo T1 da story rodar recount em prod via `pg_policies`.
5. **Pattern reference oficial:** PS RSVP commit `a84b6d9` (`20260522010000_ps_rsvp_rls_authenticated_select.sql`) — único pattern recente; minimal, idempotente, BEGIN/COMMIT. Reutilizável diretamente.

## Q1 — Ownership column em tabelas sem `user_id`

### Achado estrutural (CRÍTICO)

Conferi `students` em `20260402190833_baseline_existing_schema.sql` linhas 37-58 e TODAS as migrations subsequentes:

```sql
CREATE TABLE IF NOT EXISTS students (
  id         UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  name       TEXT NOT NULL DEFAULT '',
  phone      TEXT NOT NULL,
  cohort_id  UUID REFERENCES cohorts(id) ON DELETE SET NULL,
  is_mentor  BOOLEAN DEFAULT false,
  active     BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(phone, cohort_id)
);
```

**NÃO existe** coluna linkando `students.id` ↔ `auth.users.id`. Grep em todas as migrations confirma: nenhum `ALTER TABLE students ADD COLUMN user_id` nem coluna `auth_id`. Painel é admin-only — alunos não fazem login no Supabase Auth. Acesso a forms NPS é via **token** opaco em `nps_class_links.token`, não `auth.uid()`.

Consequência: **`USING (student_id = auth.uid())` não funciona pra nenhuma das 7 tabelas Tier 1** (sempre retorna FALSE pra qualquer linha de aluno). É a única semântica disponível pelo schema.

## Q2 — `is_dashboard_admin()`

### Versão atual em prod (presumida — única no repo)

- **Arquivo único:** `supabase/migrations/20260516020300_helper_functions.sql` linhas 25-30.
- **Body atual:**
  ```sql
  CREATE OR REPLACE FUNCTION public.is_dashboard_admin()
  RETURNS boolean
  LANGUAGE sql STABLE AS $$
    SELECT COALESCE((auth.jwt()->'user_metadata'->>'role') = 'admin', false);
  $$;
  GRANT EXECUTE ON FUNCTION public.is_dashboard_admin() TO authenticated, anon, service_role;
  ```
- **Roles/emails autorizados:** qualquer usuário cujo JWT contém `user_metadata.role = 'admin'`. **NÃO autoriza role `cs`** — vide Q1 → `response_metadata`.
- **Histórico:** grep confirmou apenas 1 `CREATE...FUNCTION is_dashboard_admin` em todo o repo. **Sem versões anteriores.** Data: 2026-05-16 (created junto com dispatch RPCs P4).
- **Usado em quê:** grep encontrou ~20 migrations subsequentes que referenciam `is_dashboard_admin()` como gate em policies/RPCs (NPS admin, dispatch, results, variants, etc.). Função é load-bearing.

### Decisão pra S.022.4

- **[x] Incluir `CREATE OR REPLACE FUNCTION` na migration** (defensivo).
- **[ ] ~~Apenas referenciar~~** — risco baixo mas custo de incluir é 6 linhas SQL.
- **Recomendação:** redefinir `is_dashboard_admin()` no topo da migration S.022.4 (antes de qualquer POLICY que a use). Razão:
  1. **Idempotência:** se função for dropada por mistake/refactor futuro, a migration S.022.4 mantém estado consistente em rerun.
  2. **Documentação inline:** quem ler a migration entende o gate sem precisar saber de qual outra migration veio.
  3. **Custo zero:** `CREATE OR REPLACE` em `STABLE` function não invalida planos pré-existentes nem invalida cache de policy quals.
- **Pre-migration smoke check (incluir no T3 ou T5):**
  ```sql
  -- 1. Função existe e tem signature esperada
  SELECT proname, pg_get_function_result(oid) AS ret
  FROM pg_proc
  WHERE proname = 'is_dashboard_admin' AND pronamespace = 'public'::regnamespace;
  -- expected: 1 row, ret='boolean'

  -- 2. Função retorna true pra admin (rodar como admin user)
  SELECT public.is_dashboard_admin();
  -- expected (auth como admin): true

  -- 3. Função retorna false pra anon
  SET ROLE anon;
  SELECT public.is_dashboard_admin();
  RESET ROLE;
  -- expected: false
  ```
- **Out-of-scope flag pra @architect:** considerar expandir `is_dashboard_admin()` pra aceitar role `cs` também (existem dashboards multi-role). Hoje cada migration cs/admin escreve inline. Não bloqueia S.022.4.

## Q3 — POLICY template reutilizável

### Pattern reference (commit `a84b6d9` — PS RSVP RLS fix)

Arquivo `20260522010000_ps_rsvp_rls_authenticated_select.sql`:

```sql
BEGIN;
CREATE POLICY "ps_rsvp_links: read for auth"
  ON public.ps_rsvp_links FOR SELECT
  TO authenticated USING (true);
CREATE POLICY "ps_rsvp_responses: read for auth"
  ON public.ps_rsvp_responses FOR SELECT
  TO authenticated USING (true);
COMMIT;
```

Pattern minimal: `BEGIN/COMMIT`, FOR SELECT, TO authenticated, USING boolean. **Sem `WITH CHECK`** (write é via service_role). Adoto isso como base + adapto por Tier.

### Tier 1 (PII sensível — admin only)

```sql
-- TEMPLATE TIER 1 — admin lê via is_dashboard_admin(), service_role bypass natural
BEGIN;

ALTER TABLE public.<table> ENABLE ROW LEVEL SECURITY;

-- DROP policies antigas com IF EXISTS pra idempotência cross-rerun
DROP POLICY IF EXISTS "<table>_select_admin" ON public.<table>;
DROP POLICY IF EXISTS "<table>_write_admin" ON public.<table>;
-- Adicionar DROPs pra policy names existentes (ver pg_policies pre-migration)

CREATE POLICY "<table>_select_admin" ON public.<table>
  FOR SELECT TO authenticated
  USING (public.is_dashboard_admin());

CREATE POLICY "<table>_write_admin" ON public.<table>
  FOR ALL TO authenticated
  USING (public.is_dashboard_admin())
  WITH CHECK (public.is_dashboard_admin());

-- Service role: bypass natural — sem POLICY explícita necessária.
-- (Se quiser explicit pra clareza, mantém pattern PS RSVP/class_nps_responses original.)

COMMIT;
```

**Exemplo concreto — `staff`:**
```sql
BEGIN;
```
