## Descrição

Como **arquiteto do EPIC-020**, preciso filtrar entries inválidas em `students` (sem email, nome=telefone, vazio) ANTES de qualquer scoring engine, porque sem cleanup o dashboard at-risk mostra 100% de falsos positivos.

**Contexto:** Spike 2026-05-05 revelou 35% das entries em `students` são lixo (54 sem email + 10 nomes-telefone + 13 vazios em cohort Advanced T2 alone). Padrão se repete nos 7 cohorts ativos. Origem: AC webhook permite registros com nome partial (capturou WA number quando nome ausente).

## Dev Notes

**Critério `is_valid_student`:** Veja ADR-017 seção "Filtro qualidade integrado".

**Não deletar entries inválidas** — operação reversível. Pode ser que AC mande update depois com nome real.

**Regex `^[0-9]+$`** detecta nomes que são puramente numéricos (telefones). Permite nomes alfa-numéricos (ex: "João 2.0").

**Casos edge a tratar:**
- Nomes muito curtos (1-2 chars): permitir? (decisão: SIM, não bloqueia)
- Email inválido formato? (decisão: validar via regex `^[^@]+@[^@]+\.[^@]+$`)
- Nome com apenas caracteres especiais (ex: "❤️", "♾️"): bloquear via regex incluir `[a-zA-Z]`?

**Estimativa volumes:** 865 students total → ~301 invalid (35%). Após cleanup, base efetiva pra EPIC-020 será ~564 alunos válidos.

## Acceptance Tests

```sql
-- Test 1: validação correta
SELECT COUNT(*) FROM students WHERE is_valid_student = true;
-- Esperado: ~564 (865 - 301 invalid)

-- Test 2: distribuição por cohort
SELECT c.name, COUNT(*) FILTER (WHERE s.is_valid_student) AS validos,
       COUNT(*) FILTER (WHERE NOT s.is_valid_student) AS invalidos
FROM students s
JOIN student_cohorts sc ON sc.student_id = s.id
JOIN cohorts c ON c.id = sc.cohort_id
WHERE c.active = true
GROUP BY c.name ORDER BY invalidos DESC;
```
