## Descrição

Como **coordenador da Academia Lendária**, preciso de uma interface no painel administrativo para enviar notificações WhatsApp para grupos e mentores, e conseguir acompanhar o status de cada envio — sem precisar acessar o banco de dados diretamente. O sistema deve funcionar de ponta a ponta: clicar "Enviar Aviso" → mensagem chega no WhatsApp em segundos.

**Contexto:** Com o schema (Story 1.1) e a Edge Function (Story 1.2) prontos, esta story fecha o ciclo: configura o Database Webhook que conecta os dois, e implementa a UI no `calendario/admin.html` para que o admin interaja com o sistema de notificações de forma intuitiva.

## Acceptance Criteria

### AC1 — Database Webhook configurado no Supabase
- **Dado** que a Edge Function `send-whatsapp` está deployada (Story 1.2)
- **Quando** o webhook `notify-whatsapp-on-pending` for criado
- **Então** deve monitorar a tabela `notifications`, evento INSERT, e invocar a Edge Function `send-whatsapp` automaticamente

### AC2 — Teste de ponta a ponta do webhook
- **Dado** que o webhook está configurado
- **Quando** um INSERT manual em `notifications` (status='pending') for feito via SQL Editor
- **Então** o status deve mudar para `processing` e depois `sent` ou `failed` em menos de 15 segundos, sem intervenção manual

### AC3 — Seção "Notificações" no painel admin
- **Dado** que um usuário com role `admin` acessa `calendario/admin.html`
- **Quando** a página carregar
- **Então** deve existir uma seção "Notificações" visível apenas para admins (verificação via JWT metadata)

### AC4 — Modal de envio de notificação
- **Dado** que o admin clica em "Enviar Aviso"
- **Quando** o modal abrir
- **Então** deve conter:
  - [ ] Seleção de tipo: `class_reminder`, `group_announcement`, `custom`
  - [ ] Seleção de classe/cohort (dropdown populado via Supabase, obrigatório para `class_reminder`)
  - [ ] Template pré-preenchido baseado no tipo selecionado (editável pelo admin)
  - [ ] Campo de mensagem livre para tipo `custom`
  - [ ] Área de preview mostrando a mensagem renderizada com as variáveis substituídas
  - [ ] Botão "Confirmar Envio" que executa o INSERT em `notifications`
  - [ ] Botão "Cancelar" que fecha o modal sem envio

### AC5 — INSERT correto em `notifications`
- **Dado** que o admin confirma o envio
- **Quando** o INSERT for executado via Supabase JS
- **Então** o registro deve ter: `type`, `class_id` (se aplicável), `target_type`, `message_template`, `status = 'pending'`, `created_by = auth.uid()`; o Database Webhook processa automaticamente

### AC6 — Histórico de notificações
- **Dado** que existem notificações registradas
- **Quando** o admin visualizar a seção de Notificações
- **Então** deve ver uma lista com: tipo, data/hora, mensagem (truncada), status com indicador visual (verde=sent, amarelo=pending/processing, vermelho=failed), opção de reenvio para status `failed`

### AC7 — Indicadores visuais de status
- **Dado** que há notificações com diferentes status
- **Quando** o histórico for exibido
- **Então** cada status deve ter cor e ícone distintos:
  - `pending` / `processing` → amarelo, ícone de relógio
  - `sent` → verde, ícone de check
  - `partial` → laranja, ícone de aviso
  - `failed` → vermelho, ícone de X

### AC8 — Reenvio de notificações com falha
- **Dado** que uma notificação tem status `failed`
- **Quando** o admin clicar em "Reenviar"
- **Então** deve fazer `UPDATE notifications SET status = 'pending', retry_count = retry_count + 1` (se `retry_count < max_retries`), acionando o webhook novamente

### AC9 — Zero regressão no fluxo de presença
- **Dado** que o painel de presença (`calendario/admin.html`) está em produção
- **Quando** a seção de Notificações for adicionada
- **Então** o calendário, o modal de presença, a navegação mensal e todas as funcionalidades existentes devem funcionar exatamente como antes

### AC10 — RLS respeitada no frontend
- **Dado** que um usuário com role `mentor` está logado
- **Quando** tentar fazer SELECT na tabela `notifications` via Supabase JS
- **Então** a query deve retornar 0 registros (RLS bloqueia no banco, não no frontend)

### AC11 — Teste real de envio (não apenas simulado)
- **Dado** que o sistema está em produção
- **Quando** o teste final for executado
- **Então** uma mensagem real deve ser enviada para o número do admin (Igor: `554399250490`) via WhatsApp, confirmando que o fluxo completo funciona em produção

## Dev Notes

### Padrão de UI do projeto
O projeto usa HTML/CSS/JS vanilla com dark mode. Verificar `calendario/admin.html` e os estilos existentes antes de criar o modal — usar o mesmo sistema de design (variáveis CSS, classes existentes de botões e modais).

### Verificação de role no frontend (padrão existente)
```javascript
const { data: { user } } = await supabase.auth.getUser();
const role = user?.user_metadata?.role;
if (role !== 'admin') {
  // ocultar seção ou redirecionar
}
```

### INSERT de notificação via Supabase JS
```javascript
const { error } = await supabase
  .from('notifications')
  .insert({
    type: selectedType,          // 'class_reminder' | 'group_announcement' | 'custom'
    class_id: selectedClassId,   // null se não aplicável
    cohort_id: selectedCohortId, // null se não aplicável
    target_type: 'both',         // ou 'group' | 'individual'
    target_group_jid: groupJid,  // JID do grupo WhatsApp
    message_template: template,
    status: 'pending',
    created_by: user.id,
  });
```

### Buscar notificações para histórico
```javascript
const { data: notifications } = await supabase
  .from('notifications')
  .select('id, type, status, message_template, created_at, sent_at, error_message')
  .order('created_at', { ascending: false })
  .limit(50);
```

### Configurar Database Webhook (via Supabase Dashboard)
1. Database > Webhooks > "Create a new hook"
2. Name: `notify-whatsapp-on-pending`
3. Table: `notifications`
4. Events: INSERT
5. Type: Supabase Edge Functions
6. Edge Function: `send-whatsapp`

### Polling de status (ADR-5: não usar Realtime)
Para atualizar status no histórico sem refresh completo, usar polling simples a cada 3 segundos enquanto houver notificações em `pending` ou `processing`.
