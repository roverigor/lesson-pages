## Story
Como **administrador da Academia Lendária**,
Eu preciso que **3 telas NPS atuais (`/admin/nps-results` + `/admin/nps-monitor` + `/admin/envios`) sejam consolidadas em 2 telas com responsabilidades claras (`/admin/dispatch` operacional + `/admin/insights` análise) + redirects 301 das URLs antigas + sidebar deduplicada**,
Para que **workflow seja linear (operacional vs análise) e PS RSVP não apareça em 2 lugares (`ps-rsvp` + aba em `nps-results`)**.

## Contexto

Discovery `03-ux-audit.md` §2.1 + §2.3 identificou **3 telas NPS + 2 lugares pra PS RSVP** servindo overlap:

### Estado atual (confuso)

| Tela | Foco | Problemas |
|------|------|-----------|
| `/admin/nps-results/` | Análise (NPS score, charts, comentários, export) | Aba "Pré PS RSVP" duplicada (também em /ps-rsvp) |
| `/admin/nps-monitor/` | Operacional (dispatcher, master switch, jobs, config) | Workflow setup → dispatch → análise quebrado em 3 telas |
| `/admin/envios/` | Auditoria (todos envios, não só NPS) | Sobreposição com `/lembretes-aulas` pra batches |
| `/admin/ps-rsvp/` | Operacional PS RSVP (envios + setup cohorts) | Análise duplicada em aba nps-results |

**Confusão observada:**
- Admin novo: "qual tela vou primeiro?"
- Workflow não-linear: setup → dispatch → análise atravessa 3 telas
- PS RSVP em 2 lugares (operacional em `/ps-rsvp` + análise em `/nps-results`)
- Sidebar tem seção "Comunicação" 2x (uma pra NPS, outra pra envios)

### Target arquitetural (2 telas)

| Tela alvo | Foco | Contém |
|-----------|------|--------|
| **`/admin/dispatch`** | Operacional | nps-monitor wizard + master switch + jobs + lembretes-aulas batch + ps-rsvp dispatch tab |
| **`/admin/insights`** | Análise | nps-results charts + nps-results aba PS → PS RSVP analytics + envios audit table |

### Princípios da consolidação

1. **Zero RPCs novos** — apenas reorganização de UI e rotas
2. **Manter dados/queries existentes** — JS modules reusados, só remontados em containers novos
3. **Redirect 301** das URLs antigas (bookmarks admin não quebram silenciosamente)
4. **Sidebar deduplicada** — 1 seção "Comunicação" com 2 sub-itens (Dispatch + Insights)
5. **Stack atual preservado** — vanilla HTML+CSS+JS (não DS v1 ainda; DS v1 sweep é story futura)
6. **Bookmarks redirect:** `/admin/nps-results` → `/admin/insights`, `/admin/nps-monitor` → `/admin/dispatch`, `/admin/envios` → `/admin/insights?tab=audit`

**Sistema EM PRODUÇÃO** mas story é **frontend interno admin** — sem impacto direto em alunos. Risco principal: muscle memory admin + bookmarks quebram (mitigado por 301).

## Acceptance Criteria

### AC1 — `/admin/dispatch` tela operacional consolidada
**Given** request `GET /admin/dispatch/`
**When** admin abre tela
**Then** carrega componentes:
- Wizard fumaça pré-dispatch (de nps-monitor)
- Master switch ativar/desativar NPS engine (de nps-monitor)
- Tabela jobs pending/recent (de nps-monitor)
- Config tunables (de nps-monitor)
- Tabela lembretes pending batch approval (de lembretes-aulas)
- PS RSVP dispatch tab (envios pendentes + setup cohorts de ps-rsvp/)
**E** layout single-page com tabs ou accordions
**E** zero RPCs novos (todos JS modules reusados)

### AC2 — `/admin/insights` tela análise consolidada
**Given** request `GET /admin/insights/`
**When** admin abre tela
**Then** carrega:
- NPS dashboard (charts, KPIs de nps-results)
- Comentários NPS + sentiment (de nps-results)
- Tabela respostas filtrada por turma/aula (de nps-results)
- PS RSVP analytics tab (KPIs + table respostas de nps-results aba PS)
- Audit table envios (de envios — tab "Audit")
- Export CSV/MD (de nps-results)

### AC3 — Redirect 301 URLs antigas
**Given** admin acessa URL antiga via bookmark
**When** request `GET /admin/nps-results/` ou `/admin/nps-monitor/` ou `/admin/envios/`
**Then** servidor (Nginx no VPS Contabo) responde `301 Moved Permanently`
**E** Location header aponta pra URL nova correspondente
**E** browser segue redirect automaticamente
**E** Old URLs documentadas como deprecated em `docs/runbooks/22.3-url-migration.md`

### AC4 — Sidebar consolidada
**Given** seção "Comunicação" atual aparece 2x em sidebar
**When** migration aplica
**Then** 1 seção "Comunicação" com 2 sub-itens:
- "Dispatch" → `/admin/dispatch`
- "Insights" → `/admin/insights`
**E** removidos links antigos (nps-results, nps-monitor, envios)
**E** PS RSVP fica em ambos (dispatch tab + insights tab) — mas link sidebar só 1x

### AC5 — Bookmarks não quebram
**Given** admin tem bookmark `/admin/nps-results/`
**When** clica bookmark
**Then** redirect 301 → `/admin/insights/`
**E** navegador atualiza bookmark automaticamente (ou prox click após)
**E** ZERO 404s por links antigos

### AC6 — Zero RPCs novos / dados preservados
**Given** consolidação é apenas frontend
**When** comparada queries SQL pré/pós
**Then** todas RPCs e queries permanecem idênticas
**E** dados em produção (NPS responses, dispatch_history, etc) acessados pelos mesmos endpoints
**E** edge functions NÃO modificadas (out-of-scope)

### AC7 — Layout responsivo dispatch
**Given** /admin/dispatch é single-page
**When** viewport varia (desktop / tablet / mobile)
**Then** tabs/accordions reorganizam-se automaticamente
**E** tabela responsive (scroll horizontal em mobile, full em desktop)
**E** master switch sempre visível (sticky top)

### AC8 — Layout responsivo insights
**Given** /admin/insights tem múltiplos charts + tabelas
**When** viewport varia
**Then** charts redimensionam (Chart.js auto)
**E** filters drawer collapse em mobile

### AC9 — Performance preservada
**Given** consolidação reusa JS modules
**When** comparado tempo first paint pré/pós
**Then** delta < 200ms (consolidação não regrida perf)
**E** lighthouse score >= score atual

### AC10 — Documentação runbook
**Given** mudança de URLs visível pra admin
**When** PR mergeado
**Then** `docs/runbooks/22.3-url-migration.md` existe contendo:
- Mapeamento URL antiga → nova
- Bookmarks admin (lista a atualizar)
- Sidebar mudanças visuais
- Rollback procedure (revert deploy + revert nginx config)

### AC11 — Smoke manual UX
**Given** consolidação live em staging/preview
**When** admin executa workflow completo
**Then** consegue: configurar dispatch → ver jobs pendentes → flip master switch → analisar resultados → export CSV — TUDO em 2 telas, sem voltar pra tela antiga
**E** documenta tempo total workflow (vs baseline 3-telas)

## Dev Notes

### Stack & ferramentas
- **Frontend stack atual:** vanilla HTML + CSS + JS (sem framework)
- **CSS base:** `/templates/admin-shared.css` (270L) + `/templates/design-tokens-dark-premium.css` (164L)
- **JS modules:** cada tela tem `app.js` inline ou separado
- **Nginx config:** `/etc/nginx/sites-available/painel-lesson-pages` na VPS Contabo
- **Deploy:** GitHub Actions automático ao push origin/main (`.github/workflows/deploy.yml`)

### Princípios reorganização

- **Não inventar componentes** — copiar JS/CSS existente
- **Tabs > accordions** quando há muitos componentes
- **Sticky elements** pra master switch + filters
- **URL params** preservados em redirect (`/admin/nps-results?cohort=T5` → `/admin/insights?tab=nps&cohort=T5`)

### Template Nginx redirect block

```nginx
# Story 22.3 — URL consolidation
location ^~ /admin/nps-results {
    return 301 /admin/insights$is_args$args;
}
location ^~ /admin/nps-monitor {
    return 301 /admin/dispatch$is_args$args;
}
location ^~ /admin/envios {
    return 301 /admin/insights?tab=audit$is_args$args;
}
```
