## O que fazer

Criar a fundação de dados do EPIC-015 no Supabase: 7 tabelas novas, extensões em 3 tabelas existentes (`students`, `survey_links`, `surveys`), RLS policies para `cs` + `admin`, e 4 jobs pg_cron (worker AC events, warmup edge functions, alert health, purge LGPD).

**Esta story destrava:** 15.B (webhook AC), 15.C (callback AC), 15.I (Meta delivery), 15.4 (refactor dispatch), 15.G (versioning), 15.L (worker function — pode ser absorvida aqui ou story separada).

## Business Value

- **Fundação técnica do épico:** sem schema + worker, nenhum fluxo automatizado AC funciona — é o gargalo absoluto da Wave 2
- **6 stories desbloqueadas em cascata** (15.B/C/I/4/G/L) = ~70h de trabalho destravado por 16h investidas
- **Article IV (No Invention):** schema rastreável a domain model existente; idempotência via DB constraint = padrão conhecido sem complexidade
- **Volume baixo (200/mês) acomodado com folga:** worker batch=50 a cada 30s tem capacidade > 100x do volume real → robustez sem complexidade prematura
- **Backward compat EPIC-004:** backfill idempotente preserva surveys legacy e dados de respostas existentes — zero risco de regressão para o painel atual
- **Quality-First Planning:** RLS embedded desde criação (não retrofit); pg_cron ativos = observabilidade operacional desde dia 1

### Schema SQL Skeleton (referência implementação)

#### `survey_versions`

```sql
CREATE TABLE IF NOT EXISTS survey_versions (
  id              UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  survey_id       UUID NOT NULL REFERENCES surveys(id) ON DELETE CASCADE,
  version_number  INT NOT NULL,
  created_at      TIMESTAMPTZ DEFAULT now(),
  UNIQUE(survey_id, version_number)
);
CREATE INDEX idx_survey_versions_survey ON survey_versions(survey_id);
```

#### `ac_purchase_events`

```sql
CREATE TABLE IF NOT EXISTS ac_purchase_events (
  id                       UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  ac_event_id              TEXT NOT NULL UNIQUE,
  payload                  JSONB NOT NULL,
  status                   TEXT NOT NULL DEFAULT 'received'
                             CHECK (status IN ('received','processing','processed','failed','duplicate')),
  processing_started_at    TIMESTAMPTZ,
  processed_at             TIMESTAMPTZ,
  retry_count              INT NOT NULL DEFAULT 0,
  last_error               TEXT,
  created_at               TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_ac_events_pending ON ac_purchase_events (created_at) WHERE status = 'received';
```

#### `pending_student_assignments`

```sql
CREATE TABLE IF NOT EXISTS pending_student_assignments (
  id                  UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  student_id          UUID NOT NULL REFERENCES students(id) ON DELETE CASCADE,
  ac_event_id         UUID REFERENCES ac_purchase_events(id) ON DELETE SET NULL,
  reason              TEXT NOT NULL,
  ac_payload          JSONB,
  suggested_cohort_id UUID REFERENCES cohorts(id) ON DELETE SET NULL,
  resolved_at         TIMESTAMPTZ,
  resolved_by         TEXT,
  resolved_cohort_id  UUID REFERENCES cohorts(id) ON DELETE SET NULL,
  created_at          TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_pending_unresolved ON pending_student_assignments(created_at) WHERE resolved_at IS NULL;
```

#### `ac_product_mappings`

```sql
CREATE TABLE IF NOT EXISTS ac_product_mappings (
  id              UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  ac_product_id   TEXT NOT NULL UNIQUE,
  cohort_id       UUID NOT NULL REFERENCES cohorts(id) ON DELETE RESTRICT,
  survey_id       UUID NOT NULL REFERENCES surveys(id) ON DELETE RESTRICT,
  template_name   TEXT NOT NULL,
  active          BOOLEAN NOT NULL DEFAULT true,
  created_at      TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_ac_mappings_active ON ac_product_mappings(active) WHERE active = true;
-- N:1: cohort_id pode repetir entre rows; ac_product_id é único
```

#### Restantes (`ac_dispatch_callbacks`, `meta_templates`)

Conforme schema em ADR-016 + spec §4.2.

### Worker Function (skeleton plpgsql)

Conforme ADR-016 §3 pseudocódigo. Considerações específicas:

- Usar `service_role` implícito (function SECURITY DEFINER opcional)
- Lock advisory para evitar concorrência se múltiplos crons sobrepostos
- Log via `RAISE NOTICE` para debug; persistir erros em `ac_purchase_events.last_error`
- Retornar `RECORD` com 2 contadores

### RLS Pattern Reuso

Adotar mesmo pattern de migrations existentes (ex: `20260408010000_surveys_schema.sql`):

```sql
ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;

DROP POLICY IF EXISTS "CS or Admin all <table>" ON <table>;
CREATE POLICY "CS or Admin all <table>" ON <table> FOR ALL TO authenticated
  USING ((auth.jwt() -> 'user_metadata' ->> 'role') IN ('admin','cs'))
  WITH CHECK ((auth.jwt() -> 'user_metadata' ->> 'role') IN ('admin','cs'));
```

### Constraints Críticas

- ❌ **NÃO modificar** estrutura existente que possa quebrar EPIC-004 — usar ADD COLUMN NULLABLE, nunca DROP/RENAME
- ❌ **NÃO criar** edge functions nesta story (15.B/C/I)
- ❌ **NÃO seedar** `ac_product_mappings` com dados reais (CS faz via UI 15.D)
- ✅ Backfill TEM que ser idempotente (rodar 2x = mesmo resultado)
- ✅ ON CONFLICT DO NOTHING em ac_purchase_events para idempotência

## Testing Strategy

### Validation Gates

| Gate | Como Validar |
|---|---|
| Schema applied | `\d+ ac_purchase_events` em psql mostra todas colunas |
| RLS funcional | Test query como cs vs admin vs anon |
| Idempotência AC | INSERT 2x mesmo ac_event_id → 1 row apenas |
| Worker function | SELECT process_ac_purchase_events_batch() retorna (0, 0) em DB vazio |
| pg_cron ativo | `SELECT * FROM cron.job WHERE jobname LIKE 'epic015-%'` |
| Backfill correto | `SELECT COUNT(*) FROM survey_links WHERE version_id IS NULL` = 0 |
| EPIC-004 não-regressão | Surveys legacy criadas antes ainda permitem dispatch + response |
| Volume baixo OK | Worker batch=50 processa < 1s em testes locais |

### Test Cases SQL

```sql
-- Idempotência
INSERT INTO ac_purchase_events (ac_event_id, payload) VALUES ('test-001', '{}'::jsonb);
INSERT INTO ac_purchase_events (ac_event_id, payload) VALUES ('test-001', '{}'::jsonb)
  ON CONFLICT (ac_event_id) DO NOTHING;
-- Verifica: SELECT COUNT(*) WHERE ac_event_id='test-001' = 1

-- N:1 mappings
INSERT INTO ac_product_mappings VALUES (..., 'PROD-A', cohort_X, survey_X, 'tpl1', true);
INSERT INTO ac_product_mappings VALUES (..., 'PROD-B', cohort_X, survey_X, 'tpl1', true);
-- Aceita: 2 produtos diferentes, mesma cohort+survey OK

-- Versioning
INSERT INTO surveys ... ;
SELECT current_version_id FROM surveys WHERE id = X;
-- Apos backfill: deve estar populado
```

### Edge Cases para Validar

- pg_cron sem `pg_net` extension instalada → warmup falha silenciosamente
- Worker tenta processar evento órfão (student deletado entre INSERT e processamento) → mark failed + retry
- Backfill em DB com 1000+ surveys legacy → tempo razoável (<10s)

### Não-Testáveis Nesta Story

- Worker realmente disparar dispatch (Story 15.B integra)
- Slack alerts via `alert_slack_if_unhealthy()` (Story 15.E impl real)
- Performance sob 200/mês carga real (validar em prod pós 15.B)
