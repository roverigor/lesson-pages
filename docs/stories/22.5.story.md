## Story
Como **administrador da Academia Lendária**,
Eu preciso que **3 webhooks de compra (`ac-purchase-webhook` + `hotmart-purchase-webhook` + `generic-purchase-webhook`) tenham precedência clara documentada (AC > Hotmart > Generic) + coluna `source` enum em `ac_purchase_events` + dedup por tupla `(email, product_id, purchase_date)`**,
Para que **mesmo evento via 2 webhooks gere 1 linha (não duplicates) e fica claro qual ganha em conflito**.

## Contexto

Discovery `01-architecture-map.md` §4.3 identificou **3 webhooks compra em produção** sem precedência formal:

| Webhook | Fonte | Tabela | Status |
|---------|-------|--------|--------|
| `ac-purchase-webhook` | ActiveCampaign | `ac_purchase_events` | Ativo — EPIC-015 |
| `hotmart-purchase-webhook` | Hotmart | `ac_purchase_events` (mesma) | Ativo |
| `generic-purchase-webhook` | Fallback | `ac_purchase_events` (mesma) | Ativo |

**Problemas:**
- Se Hotmart event fire `hotmart-*` E `generic-*` → 2 linhas pra mesma compra
- AC e Hotmart podem disparar simultaneamente pro mesmo aluno → race condition
- Coluna `source` não existe → impossível auditar qual webhook escreveu
- Sem dedup tupla → mesma compra pode aparecer N vezes em `student_imports`

**Decisão arquitetural:**
- **AC = canonical** (source-of-truth contratual — AL contrata AC pra automation)
- **Hotmart = fallback** (eventos sem counterpart AC)
- **Generic = deprecated** (logging-only por 30d, depois 410 Gone)
- Coluna `source` enum `('ac'|'hotmart'|'generic')` em `ac_purchase_events`
- Dedup por tupla `(email, product_id, purchase_date)` — UNIQUE constraint

## Acceptance Criteria

### AC1 — Coluna `source` enum em `ac_purchase_events`
**Given** schema atual sem coluna source
**When** migration aplica
**Then** `ALTER TABLE ac_purchase_events ADD COLUMN source text` com CHECK `source IN ('ac','hotmart','generic')`
**E** rows existentes recebem `source = 'ac'` (assumir default — origem mais comum)
**E** índice `idx_ac_purchase_events_source` criado

### AC2 — Dedup tupla UNIQUE constraint
**Given** mesma compra dispara 2 webhooks
**When** ambos tentam INSERT
**Then** UNIQUE `(email, product_id, purchase_date)` rejeita o segundo
**E** primeiro webhook chamado wins (race based)
**E** segundo recebe response indicando "duplicate, no-op"
**E** audit_log entry tipo `purchase_dedup_blocked` com source rejected

### AC3 — Precedência documentada via webhook behavior
**Given** mesmo evento (mesmo email + product + date) chega via AC E Hotmart
**When** ambos processam
**Then** AC sempre wins (precedência via timing OR explicit check)
**E** se Hotmart chegou primeiro, AC pode UPSERT (UPDATE existing row marcando `source='ac'` se source era 'hotmart')
**E** Generic NUNCA escreve se AC ou Hotmart já escreveu — apenas logs

### AC4 — Generic webhook deprecated (30d safety)
**Given** generic-purchase-webhook ainda receber requests
**When** migration aplica
**Then** generic responde 200 OK + logs payload em `audit_log` (não INSERT em ac_purchase_events)
**E** 30 dias após apply, generic retorna 410 Gone (story follow-up 22.5.cleanup pra 2026-06-22)
**E** Slack alert se generic receber > 0 requests em 7d (anomalia)

### AC5 — Backfill source em rows existentes
**Given** ~N rows pré-existentes sem source
**When** migration aplica
**Then** UPDATE `source = 'ac'` em todas WHERE source IS NULL
**E** backfill é idempotente (rerun não muda nada)

### AC6 — ADR-023 webhook precedence
**Given** decisão arquitetural
**When** PR mergeado
**Then** `docs/architecture/ADR-023-webhook-purchase-precedence.md` existe contendo:
- Decisão AC > Hotmart > Generic
- Razão (AC contratual source-of-truth)
- Alternativas (todos igual, last-wins, first-wins) rejeitadas
- Deprecation timeline generic (30d)
- Dedup tupla rationale

### AC7 — Smoke test triplo
**Given** local PG 15
**When** dispara 3 webhooks com mesmo (email, product, date)
**Then** ac-purchase chega → INSERT row
**E** hotmart-purchase chega → tenta INSERT, UNIQUE bloqueia, response no-op
**E** generic-purchase chega → log only, sem INSERT
**E** count rows = 1
**E** source = 'ac'

### Template migration ADD source + UNIQUE

```sql
-- T2 partial
ALTER TABLE public.ac_purchase_events
  ADD COLUMN IF NOT EXISTS source text DEFAULT 'ac';

ALTER TABLE public.ac_purchase_events
  ADD CONSTRAINT IF NOT EXISTS chk_ac_purchase_source
  CHECK (source IN ('ac','hotmart','generic'));

UPDATE public.ac_purchase_events
   SET source = 'ac'
 WHERE source IS NULL;

ALTER TABLE public.ac_purchase_events
  ALTER COLUMN source SET NOT NULL;

-- Gate T1: dedup query = 0

  ADD CONSTRAINT IF NOT EXISTS ac_purchase_events_dedup_unique
```
